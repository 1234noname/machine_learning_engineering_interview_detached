# Database - Postgres + pgvector

The persistence layer is a single PostgreSQL 16 database with the `pgvector` extension, holding the Fashion200k catalog with two embedding columns and the conversational session history. The choice over a dedicated vector store is operational: one engine for relational + vector, transactional updates across product metadata and embeddings, and at ~5k–200k catalog rows the recall-vs-cost trade comfortably lands on a single Postgres instance rather than a managed ANN service.

Every behavioural claim is tied to a file in `specs/db/`, `infra/migrations/`, or the orchestrator's `retrieval_tool.ex`; known drift between the declared schema and what's actually wired is named in §3.

---

## 1. What's stored

Two schemas, four tables, two HNSW indexes:

| Schema / table | Role | Defined in |
|---|---|---|
| `catalog.products` | Fashion200k product catalog - title, attributes, price, two embedding vectors | `specs/db/catalog.sql` |
| `conversations.conversations` | Per-session outer record with `expires_at` for TTL | `specs/db/conversations.sql` |
| `conversations.turns` | Append-only per-turn history with JSONB content; FK → conversations with `ON DELETE CASCADE` | `specs/db/conversations.sql` |
| `public.schema_migrations` | Migration ledger written by `infra/migrations/migrate.sh` | `infra/migrations/migrate.sh:18-21` |

The schema sources in `specs/db/` are the canonical declarations; the migrations under `infra/migrations/` re-use them via psql `\ir` (`include-relative`) rather than duplicating SQL. That keeps "what the table looks like" in one file and "how it got there" in another (§2iii).

The remainder of the doc covers the catalog and conversation schemas in turn (§2i–§2ii), the spec/migration discipline that keeps them honest (§2iii), the query and pooling layers (§2iv–§2vi), the deployment posture across two Postgres backends (§2vii), the ingestion path (§2viii), and the gaps the deployed shape doesn't yet cover (§3).

## 2. Discussion

### 2i. Catalog - one wide row, two HNSW indexes

`catalog.products` (`specs/db/catalog.sql:22-34`) is a single wide row, not a star schema:

```sql
CREATE TABLE catalog.products (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    category        TEXT NOT NULL,
    colour          TEXT NOT NULL,
    formality       TEXT NOT NULL,
    occasion        TEXT NOT NULL,
    price_cents     INTEGER NOT NULL,
    image_url       TEXT NOT NULL,
    embedding       vector(768) NOT NULL,
    text_embedding  vector(512),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

The four attribute columns (`category`, `colour`, `formality`, `occasion`) match the four-key MCP attribute object documented in [3.MCP.md](./3.MCP.md) §2vii - the same shape the LLM and ViT attribute head produce, persisted verbatim so the kNN colour constraint (§2iv) can be a single equality predicate.

**`embedding` is `NOT NULL`; `text_embedding` is nullable.** Every product must have a 768-d image embedding (the catalog cannot be ingested without one). The 512-d text embedding is optional - Fashion200k products without usable text descriptions still ingest, and the text kNN query explicitly filters them out (§2iv). The nullable text column was the lever that let migration 005 land additively without rejecting existing rows.

**Two HNSW indexes, both cosine.** `products_embedding_hnsw` and `products_text_embedding_hnsw`, each `USING hnsw (... vector_cosine_ops)`. The choice of cosine over L2 is forced by the upstream encoders: ViT and CLIP both return L2-normalised embeddings (per [4.API.md](./4.API.md) §2v and the batcher's defensive re-normalisation in `batch_queue.rs:267`), and the `<=>` cosine operator gives the correct similarity ranking on unit vectors. HNSW over IVFFlat for both recall and query latency; IVFFlat's advantage is lower memory and faster build time, which at this corpus size and update cadence isn't load-bearing. The two indexes are created without explicit `m` / `ef_construction` parameters - the pgvector defaults (`m=16, ef_construction=64`) apply.

**The pgvector version guard the schema comment claims is not wired.** The schema comment names `pgvector ≥ 0.5.0` as the version requirement (HNSW was added then) and refers to a CI `spec-validate-sql` job that would pin a pgvector-enabled Postgres image to satisfy it. That job does not exist (no matching workflow in `.github/workflows/`). Wiring it would be the natural mechanical guard against a pgvector version regression; until then the version requirement rests on developer attention.

**Index maintenance posture is unmanaged.** pgvector has no online HNSW rebuild, and HNSW indexes can fragment under sustained churn. At the current ingestion shape (bulk catalog seeds, no high-velocity row updates) this isn't a binding concern, but a future high-churn workload would need a `REINDEX` cadence the deployment doesn't currently schedule.

### 2ii. Conversations - flat history, TTL on the outer row

`conversations.conversations` and `conversations.turns` (`specs/db/conversations.sql:15-28`) are the durable side of the per-session GenServer state the orchestrator holds in memory:

- The outer `conversations` row carries `id` and `expires_at` only - created at the start of a chat session by `AVSA.ConversationRecord.insert_conversation/1`, which sets `expires_at = now() + 24h` (`conversation_record.ex:45`).
- Each user/assistant/tool turn appends to `conversations.turns` via `AVSA.ConversationRecord.insert_turn/3`. The `role` column has a `CHECK (role IN ('user', 'assistant', 'tool'))` enum at the table level; the `content` column is JSONB so the message envelope can evolve without a migration.
- `ON DELETE CASCADE` on the FK means dropping a conversation drops every turn it held - single-statement delete, no orphan rows possible.

**The TTL posture is deliberate: predicate in application code, no `pg_cron`.** The schema's own comment names it: *"a periodic cleanup job (cron / scheduled worker) deletes rows where `expires_at < now()`. No trigger and no pg_cron dependency - the cleanup predicate lives in application code so it's testable and observable, and we don't take a hard dep on a pg extension that isn't universal."* The reasoning is portability - Cloud SQL and Neon both vary in their `pg_cron` posture, and the application-code path runs the same way against any Postgres. The cleanup job itself doesn't exist today — that gap is owned by §3.

**All writes are best-effort, and failures are silently swallowed.** Both `insert_conversation/1` and `insert_turn/3` guard on `repo_running?()` and return `:ok` silently if the Ecto repo isn't started — the same code path runs in stub-mode tests where no Postgres is configured. When the repo IS running, both functions catch `{:error, _}` from `AVSA.Repo.insert` and return `:ok` without logging or emitting telemetry (`conversation_record.ex:52, :68`). A failed DB write produces nothing observable from outside the function; the chat flow continues even if the durable history is unavailable. This matches the design intent — a flaky DB write should not surface to the user mid-turn — but it also means a sustained DB outage produces no operational signal from this code path.

**Only `user` turns are written today.** The schema's `role` enum allows `user`, `assistant`, and `tool`, but the orchestrator's `commit_turn/5` (`conversation.ex:185-189`) calls `insert_turn/3` exactly once per chat turn with `role = "user"` and `content = %{text: user_text, results_count: length(results)}`. Assistant responses and tool-call payloads are not currently persisted — the JSONB shape supports it, the call site doesn't yet. A future "show me my last conversation" surface would need that write path added and a content schema agreed on; for now `conversations.turns` is a user-utterance log with the result count, not a full transcript.

### 2iii. Spec ↔ migration split, and the runner

Two files describe each schema: the spec under `specs/db/` (the canonical, declarative version) and the migration under `infra/migrations/` (the imperative apply-this-once version). The split exists because the spec is the contract - `specs/db/catalog.sql` is what the ingest job, the kNN query, and the test fixtures all agree on. Duplicating its DDL into a migration would create a second source that could drift. So migration 001 reads:

```sql
\ir ../../specs/db/catalog.sql
```

Migration 002 does the same for the conversations spec. Migrations 004, 005, and 006 are actual DDL - they're *changes* on top of the spec, not the spec itself. This is also why migrations 001 and 002 are not internally idempotent (`CREATE TABLE` / `CREATE INDEX` with no `IF NOT EXISTS`): idempotency is the runner's job, not the spec's.

**The runner (`infra/migrations/migrate.sh`) maintains `public.schema_migrations` as a ledger.** Each `[0-9]*.sql` under the migrations directory is applied in lexical order; the runner records the filename in the ledger and skips files already recorded. The crucial property: each migration runs under `psql --single-transaction` together with its ledger insert, so a migration that fails halfway never records as applied (`migrate.sh:33-37`). Re-running `just db-migrate` after a failure re-attempts the same file from the top.

**The runner is forward-only; there is no `migrate.sh down` equivalent.** A bad migration is corrected by writing a new migration that undoes the change (the migration 003 → 006 pair below is exactly that pattern), not by rolling back the ledger. The trade is that schema evolution is an append-only history — easier to reason about and audit, harder to walk back from a mistake. For any change with a wide blast radius, the discipline is to validate against a staging database before running against production.

**Migration 003 is intentionally absent and the cause is documented in 006.** Migration 003 originally created an `audit` schema that backed LLM-spend / budget tracking (`BudgetQuery.GetDailySpend`) and `mcp_tool_call` rows from the now-deprecated MCP gateway. When the gateway consolidated onto the JSON-RPC MCP server (which does not audit - see [3.MCP.md](./3.MCP.md) §1), nothing wrote or read those tables any more. Migration 003 was removed from the migrations directory and migration 006 added with `DROP SCHEMA IF EXISTS audit CASCADE` to clean up databases that had already run 003. Fresh databases see no `audit` schema; existing databases lose it on the next `just db-migrate`. The gap in the lexical sequence is a durable artefact of the rollback, not a missing file.

Migrations 004 and 005 each carry a guard pattern (`DO $$ BEGIN IF EXISTS / NOT EXISTS ... END $$`) so they're no-ops on fresh databases (where the current spec already has the desired state) and corrective on databases that pre-date the spec change. Migration 004 renames `price_pence` to `price_cents` — the catalog stores ZAR prices (the rand's minor unit is cents); *"pence"* was a British holdover from the initial schema draft. The catalog API hardcodes `_CURRENCY = "ZAR"` (`catalog.py:34`) to match.

### 2iv. The kNN queries - three variants, one operator

`AVSA.RetrievalTool` (`apps/orchestrator/apps/avsa/lib/avsa/retrieval_tool.ex:10-46`) defines three pre-parameterised SQL queries as module attributes:

| Query | Used when | Parameter shape |
|---|---|---|
| `@knn_sql` | Image kNN, no colour constraint | `($1=embedding, $2=prior_ids[])` |
| `@knn_colour_sql` | Image kNN + colour constraint | `($1=embedding, $2=prior_ids[], $3=colour)` |
| `@knn_text_sql` | Text kNN | `($1=text_embedding, $2=prior_ids[])` |

All three:

- Use the `<=>` cosine-distance operator: `embedding <=> $1` in the `ORDER BY` and as the `score` projection (smaller is closer).
- Apply the `prior_result_ids` exclusion via `WHERE id != ALL($2::uuid[])` - past results from the same conversation are filtered out so a follow-up turn doesn't re-surface what the user already saw (the mechanism `3.MCP.md` §2vii describes from the agent side).
- `LIMIT 20` - the kNN returns at most 20 candidates per call.
- Use parameterised binding for every value, including the colour constraint (case-insensitive via `lower(colour) = lower($3)` so the comparison happens server-side; the user's text *"navy"* matches a row stored as *"Navy"* without any client-side normalisation).

The text kNN additionally carries `WHERE text_embedding IS NOT NULL` - products that ingested without a text description are silently excluded from text search. This is consistent with the schema's nullability but worth knowing: a text query against a catalog where many rows have no text embedding will see a smaller effective corpus.

The latency budget is config-driven: `[latency] retrieval_knn_ms = 150` in `config/avsa.base.toml`, and `RetrievalTool` logs a warning if the query exceeds it. The threshold is an early-alert signal for index degradation or table growth, not a binding constraint at the deployed corpus size.

### 2v. The browse query - offset pagination, naive count

The catalog browse endpoint (`apps/api/src/avsa_api/routes/catalog.py:40-44`) is two simple statements:

```python
_PAGE_QUERY = (
    "SELECT id, title, category, price_cents, image_url "
    "FROM catalog.products ORDER BY id LIMIT $1 OFFSET $2"
)
_COUNT_QUERY = "SELECT count(*) FROM catalog.products"
```

Ordering by `id` (the UUID primary key) gives stable pagination - re-running the same page yields the same rows in the same order regardless of insert activity. `LIMIT $1 OFFSET $2` is the simplest possible pagination shape and matches the OpenAPI contract in [4.API.md](./4.API.md) §2i.

Both queries are honest about their scale envelope. `OFFSET N` is `O(N)` on every page request: Postgres still has to walk the index past N rows before yielding the page. `count(*)` needs a full sequential scan — the HNSW indexes are similarity indexes; no index in this schema can shortcut a global row count. At ~5k rows (the synthetic-v1 default seed) both are sub-millisecond; at the full Fashion200k load (~200k+ rows) the count becomes noticeable and deep pages slow. §3 names this as a gap with a concrete fix (keyset pagination + a counter materialised view, or accepting *"exact count"* as a non-goal and returning the page without `total`).

### 2vi. Connection pooling

Two pools, each owned by the language that uses them.

**Python (api → catalog reads).** `apps/api/src/avsa_api/main.py` opens an `asyncpg` pool at app startup (`lifespan` async generator):

```python
db_pool = await asyncpg.create_pool(db_url)
```

No explicit `min_size` / `max_size` - `asyncpg` defaults (`min_size=10`, `max_size=10`) apply. When `AVSA_DB_URL` is unset the pool is `None` and the catalog routes return 503 with a structured `{"code": "catalog_unavailable"}` error rather than crashing.

**Elixir (orchestrator → catalog kNN and conversation writes).** `AVSA.Repo` (`apps/orchestrator/apps/avsa/lib/avsa/repo.ex`) is a standard `Ecto.Repo` with the Postgres adapter; the pool size is config-driven via `[db.pool] size = 10` in `config/avsa.base.toml`. Ecto's connection pooling is per-process and built on `db_connection`.

**A `[db]` section in `config/avsa.base.toml` declares SQLAlchemy-style pool sizing** (`pool_size = 5`, `max_overflow = 10`) - but the comment in the file says explicitly *"no code reads these yet"*. The settings are pre-positioned for a future SQLAlchemy-using consumer; today they're documentation, not active configuration (see §3).

### 2vii. Deployment posture

The deployed system runs against two Postgres backends — Cloud SQL on the GKE path, Neon on the Modal `api_app` demo path — per [1.ARCHITECTURE.md](./1.ARCHITECTURE.md) §2v. They are not synchronised; this doc describes the schema shape both target, but a write through one path is not visible from the other.

**Backups inherit each provider's defaults.** Cloud SQL gives automated daily backups + point-in-time recovery within the configured retention window; Neon's branching model preserves snapshots as branches. Neither path is explicitly configured beyond what comes out of the box — acceptable for the current operational shape, but the kind of thing to review before either path takes paying users.

**TLS is provider-mandatory but application-side enforcement is missing.** Both Cloud SQL and Neon require TLS for client connections, and `asyncpg` / Ecto honour `sslmode=require` in the connection string. Whether `DATABASE_URL` actually carries `sslmode=require` in any deployed environment rests on whoever set the env var — there is no application-side check that refuses to start without it. A boot-time guard (*"refuse to start if `DATABASE_URL` lacks `sslmode=require` and the host is not localhost"*) would close this drift surface; it does not exist today.

**Query-level observability is unconfigured at the application layer.** Slow-query logging, autovacuum tuning, lock contention surfacing, and `pg_stat_statements` are not explicitly set up — whatever Cloud SQL or Neon expose through their own consoles is what's available. Wiring `pg_stat_statements` into the Prometheus scrape (via `postgres_exporter` or equivalent) is the natural extension if a query-level observability surface becomes load-bearing.

### 2viii. Catalog ingestion

`scripts/seed-catalog.py` is the canonical seeder, invoked via `just seed-catalog` against a database that has already run `just db-migrate`. Two data sources, selected by `[catalog.source]` in the generated `config/avsa.toml`:

- **`synthetic-v1`** (default, hermetic): deterministic per-row synthetic products generated by `catalog_seed`. Titles, colours, prices, image URLs, and 768-d embeddings all derive from a fixed-seed RNG. The seed job is reproducible - running it twice on a fresh database produces byte-identical rows. This source keeps CI and the hermetic pytest fixture working without external data.
- **`fashion200k`** (opt-in, primarily for local quality runs against the real corpus): joins a self-describing manifest to a pre-computed embedding artifact (`data/embeddings/<sha256>`) and loads real Fashion200k titles, derived colour/formality/occasion attrs, real 768-d image embeddings, and 512-d text embeddings into `catalog.products`. **The seeder does not call the live ViT per row** - embeddings come from the offline artifact built once by the embedding pipeline in `libs/avsa-data/`. The catalog ingest is decoupled from the model service for exactly this reason: a full Fashion200k load would otherwise hit the ViT ~200k times.

The CI / hermetic-test path is always `synthetic-v1`; `fashion200k` is opted into via the config overlay during local quality work or deliberate production seeding.

**Throughput is `COPY ... FROM STDIN`.** The seeder uses psycopg3's `COPY` protocol rather than row-by-row `INSERT` - the script's own comment notes this is *~10× faster at 5000+ rows and gates the CI seed budget*. The embedding vectors are encoded in pgvector's text representation (`[0.1,0.2,...]`) — pgvector accepts both text and binary on COPY, and the seeder picks text for code simplicity; the throughput budget is met without the binary path.

For the `synthetic-v1` path, embeddings can be either stub-generated (deterministic from the row's index) or fetched live from the model `/embed` endpoint, controlled by `AVSA_EMBED_STUB`. The `fashion200k` path ignores that flag - the artifact is authoritative.

## 3. Three pertinent gaps

Three places where the deployed database posture is honest about what's not wired today. Each is named with the design intent, the current posture, and the concrete fix.

**1. The conversation TTL has no scheduled cleanup job.** Design intent (from `specs/db/conversations.sql:7-10`): a periodic application-code job deletes rows where `expires_at < now()`. Current posture: no such job exists in the orchestrator's supervision tree (`apps/orchestrator/apps/avsa/lib/avsa/`), in `scripts/`, or as a CronJob in `helm/templates/`. Conversations created today live forever until the database is reset. The fix is small - an `AVSA.ConversationCleanup` GenServer that runs every N minutes via `Process.send_after/3` and issues `DELETE FROM conversations.conversations WHERE expires_at < now()` (the `ON DELETE CASCADE` on `conversations.turns` does the rest). Wiring it into `AVSA.Application.start/2`'s children list is one line. Until that lands, the TTL column is bookkeeping for a job that doesn't run.

**2. Catalog browse scales linearly with depth and with row count.** `OFFSET N` walks past `N` rows on every page; `count(*)` scans the whole table. At the synthetic-v1 default (~5k rows) both are sub-millisecond; at the full Fashion200k load (~200k+ rows) the deep-page case becomes noticeable and the count grows linearly with the catalog. Two fixes, depending on what the API contract needs to keep:

- *If `total` is essential:* keep `_COUNT_QUERY` but pair it with a counter materialised view refreshed on a schedule (or via an `AFTER INSERT/DELETE` trigger), and switch the pagination to **keyset** - `WHERE id > $last_id ORDER BY id LIMIT $limit`. Constant time per page regardless of depth.
- *If `total` can drop:* remove the count and return only `items` + a `has_more` boolean derived from `LIMIT $limit + 1`. The browse grid in [5.FRONTEND.md](./5.FRONTEND.md) §2iv currently uses `total` to compute total pages - it would adapt to "next button enabled iff `has_more`" without much rework.

The cleanest evolution is the second (counts at this scale are rarely worth their cost), but the contract change has to land in [4.API.md](./4.API.md) §2i and the OpenAPI spec first.

**3. The Python asyncpg pool sizing is implicit; the documented `[db]` section is not wired.** Two related items. The `asyncpg.create_pool(db_url)` call in `apps/api/src/avsa_api/main.py` does not pass `min_size` / `max_size` - the defaults (`10`/`10`) apply silently, which is fine for the current concurrency but not visible in config. Separately, `config/avsa.base.toml` carries a `[db]` section with SQLAlchemy-style `pool_size` / `max_overflow` / `pool_recycle` that the source comment says no code reads. The two-pool reality (Python `asyncpg` defaults; Elixir `Ecto` `[db.pool] size = 10`) is correct; the config layer suggests something more elaborate that isn't there. The fix is small: drop the `[db]` SQLAlchemy block from the config until something actually consumes it, add a `[db.asyncpg]` section that `main.py` reads explicitly, and document both pools in one place. Until that lands, the config gives the wrong shape of the truth.
