# Frontend - The Shopper App

The shopper is a Next.js 15 app under `frontend/apps/shopper/`. It serves one user-facing surface - a single page combining image+text search, a catalog browse grid, and a status header with a live API health badge - plus eight server-side route handlers that front every public-facing API call. The app's job is small but specific: own the user experience for AVSA's visual shopping flow, and keep the API host off the client bundle.

---

## 1. What the app is

A pnpm workspace under `frontend/`, one app and one shared package:

| Package | Path | Role |
|---|---|---|
| `@avsa/shopper` | `frontend/apps/shopper/` | The Next.js 15 user-facing app |
| `@avsa/shared` | `frontend/packages/shared/` | TS types generated from `specs/api/chat.openapi.yaml` via `openapi-typescript` |

The shopper itself is small: one user-facing page, five components, five library modules, eight server route handlers.

**Routes and layout (`app/`):**

| Path | Role |
|---|---|
| `app/page.tsx` | The single user-facing page - chat input + result stream + browse grid + header |
| `app/layout.tsx` | Root layout: HTML shell, Inter font via `next/font/google`, `globals.css` |
| `app/chat/route.ts` | `POST /chat` - multipart + SSE proxy to the API service |
| `app/mcp/route.ts` | `POST /mcp` - JSON-RPC body verbatim to the orchestrator's MCP server |
| `app/images/[...path]/route.ts` | `GET /images/{path}` - signed-image proxy to the API |
| `app/openapi.json/route.ts` | `GET /openapi.json` - passes the API's OpenAPI spec at root for the Swagger UI |
| `app/api/catalog/route.ts` | `GET /api/catalog` - paginated catalog proxy |
| `app/api/health/route.ts` | `GET /api/health` - API liveness proxy |
| `app/api/docs/route.ts` | `GET /api/docs` - proxies the FastAPI Swagger UI |
| `app/api/metrics/route.ts` | `GET /api/metrics` - Prometheus exposition for `avsa_shopper_*` metrics |

**Components and library modules:**

| Path | Role |
|---|---|
| `app/components/ChatInput.tsx` | Multipart upload form + SSE-consumer driver |
| `app/components/MessageStream.tsx` | Renders the product-card grid produced by a chat turn |
| `app/components/BrowseGrid.tsx` | Paginated catalog browse with retry |
| `app/components/ProductCard.tsx` | Presentational product card - used by both message stream and browse grid; intentionally non-interactive since the product detail side-panel was removed in a UX revision |
| `app/components/HealthBadge.tsx` | Live API-reachability indicator in the page header |
| `app/lib/consumeSse.ts` | Reusable SSE-frame consumer driven by `ChatInput` |
| `app/lib/imageProxy.ts` | URL validation seam for the signed-image proxy chain |
| `app/lib/imageRemotePatterns.mjs` | Single-source-of-truth host policy shared with `next.config.js` |
| `app/lib/metrics.ts` | Prometheus registry pinned on `globalThis` to survive hot reload |
| `app/lib/nextConfig.ts` | Next config builder importing the shared host policy |

The stack is intentionally narrow: Next 15.5, React 18.3, TypeScript 5.4, plus `prom-client` for instrumentation and `@avsa/shared` for wire types. No state-management library, no UI component library, no data-fetching library - pure React state and `fetch` everywhere. Build is `output: "standalone"` for self-contained Node deployment.

## 2. Discussion

### 2i. Type contracts from the OpenAPI spec

The `@avsa/shared` package exists to make UI-vs-API contract drift mechanically impossible. Its `package.json` carries a single `generate` script - `openapi-typescript ../../../specs/api/chat.openapi.yaml -o src/generated.ts` - that regenerates `src/generated.ts` from the OpenAPI spec. The package's `index.ts` re-exports `paths`, `components`, `operations`, `$defs`, and `webhooks` from the generated file with a header comment: *"DO NOT hand-write types here. All types are derived from `specs/api/chat.openapi.yaml`."*

Consumers in the shopper import the schemas they need:

```tsx
import type { components } from "@avsa/shared";
type ChatProductCard = components["schemas"]["ProductCard"];
type CatalogPage    = components["schemas"]["CatalogPage"];
```

The same `ProductCard` shape the API documents in its OpenAPI spec is the type the React components declare against. If the OpenAPI changes - adding a required field, renaming `image_url` to `imageUrl`, dropping `category` - the TS build breaks at every component that consumed the old shape. There is no "remember to update the frontend types" step; the regeneration is the contract sync.

The cost is that every API spec change requires re-running `pnpm --filter @avsa/shared generate` and committing the regenerated `src/generated.ts`. The trade is worth it because the alternative - hand-maintained UI types - is exactly the kind of slow-drift surface where a missed field becomes a runtime `undefined` in production.

### 2ii. Server-only proxies

`AVSA_API_URL` is deliberately a server-only env var (no `NEXT_PUBLIC_` prefix). The browser never knows where the API lives; it always fetches from the shopper's origin. Every public-facing API call routes through a Next.js server route handler that reads `AVSA_API_URL` at request time and proxies upstream:

- `/chat` → `$AVSA_API_URL/chat` (multipart + SSE pump)
- `/mcp` → `$AVSA_MCP_URL` (JSON-RPC body verbatim)
- `/images/{path}` → `$AVSA_API_URL/images/{path}` (with `token` and `expires` forwarded)
- `/api/catalog`, `/api/health`, `/api/docs`, `/openapi.json` → corresponding API endpoints

There are two operational consequences: first, CORS never enters the picture: the browser fetches same-origin everywhere. Second, the API host (a ClusterIP inside GKE) stays invisible to anything that doesn't run server-side in the shopper. Local development, staging, and prod can target different API hosts without touching the client bundle.

Catalog rows store `image_url` as a tokenless relative `/images/{key}`; the catalog route (`apps/api/src/avsa_api/routes/catalog.py`) signs it at read time into `/images/{key}?token=…&expires=…`. The shopper's `<Image>` component renders that relative URL, which the browser resolves against the shopper's own origin. The `app/images/[...path]/route.ts` handler picks it up, forwards only the `token` and `expires` params (drops anything else as a defensive measure), and proxies to the API. End-to-end, the browser only ever sees the shopper origin; the API origin never appears in the rendered DOM or any network panel.

Next.js's `<Image>` enforces `images.remotePatterns` at build time; absolute image URLs from disallowed hosts won't render. The shopper's runtime image proxy (`app/lib/imageProxy.ts`) independently validates host + path before passing a URL to `<Image>`. Both consumers - `next.config.js` (the build-time enforcement) and `imageProxy.ts` (the runtime validation) - import their host policy from `app/lib/imageRemotePatterns.mjs`, a plain ESM module that exports `buildRemotePatterns()` and `canonicalOrigins()`. The choice of `.mjs` matters: `next.config.js` can't import a `.ts` file without a transpile step, so the policy lives in a format both can consume directly. The Next allowlist and the proxy validation cannot drift; they read the same constants.

### 2iii. The chat surface

`ChatInput` (`app/components/ChatInput.tsx`) is the input form and the driver of the SSE consumer. The interesting mechanics:

The form accepts JPEG, PNG, WebP, and HEIC (matching the API's MIME allowlist in `5.API.md` §2i), with a 10 MB per-file cap. The client cap (`MAX_FILE_SIZE` in `ChatInput.tsx`) and the server's `max_upload_bytes` (in `config/avsa.base.toml`) are independent values that happen to be equal today - same shape for the MIME allowlist (`ACCEPTED_TYPES` in the component vs `ALLOWED_MIME` in the API). Neither has a mechanical guard against drift; if the server moves, the client should be updated to match. Files can be added via file picker, drag-and-drop, or pasted URL (the paste handler fetches the URL and adds the resulting blob). Each staged upload gets an `ObjectURL` for thumbnail preview; the URL is revoked on item removal and on unmount via the cleanup effect, so previews don't leak memory across multi-turn sessions.

Submitting a new search aborts the previous fetch (`abortRef.current?.abort()`). This matters because chat turns can take ~1–2 seconds (the round-trip described in `3.QPS.md` §2i); a user typing a refinement query before the previous response completes would otherwise stack two SSE streams.

`consumeSse` (`app/lib/consumeSse.ts`) is a ~40-line utility: `TextDecoder` over the response's `ReadableStream`, line-buffered, accumulating partial chunks across reads. For each `data: ` line, it `JSON.parse`s the payload and invokes the caller's event handler. Malformed frames are silently skipped (`try/catch` around the parse) so a single bad event doesn't tear down the whole stream. The function is exported separately from `ChatInput` so unit tests can drive it directly without rendering a React component.

The SSE wire shape is documented in `5.API.md` §2i: bare `data: <json>\n\n` lines, no `event:` / `id:` / `retry:` fields, exactly one variant (`product_card`) today. The consumer's design - switch on `event.type` - is forward-compatible: adding a `chat_error` variant (proposed in `5.API.md` §3) wouldn't require any change to the consumer interface.

The page holds the session ID in `useState`; `ChatInput` sends it as `X-Resume-Conversation-Id` on the next request and reads `X-Conversation-Id` from the response. Inside the orchestrator the conversation_id is the lookup key for the per-session GenServer; `prior_result_ids` from prior turns flow into the next kNN as an exclusion (see `4.MCP.md` §2vii). On the UI side this means a follow-up query like `"show me more like the first one but in navy"` re-retrieves with the already-seen IDs filtered out - surfaced as "different products this turn," not "duplicates of last turn."

The session ID is also exposed as a copyable affordance at the bottom of the page body, below the browse grid (*"Session ID: <uuid> [Copy]"*) - useful for debugging a flow against logs. §3 flags this as a deliberate debug surface that would not survive a production hardening pass.

### 2iv. The browse surface

`BrowseGrid` (`app/components/BrowseGrid.tsx`) is a client component that fetches `/api/catalog?page=N` on mount and on page change. The interesting bits are how it handles the server's clamped-limit contract and how it implements retry.

The API contract (`5.API.md` §2i) clamps `limit` to a configured ceiling - an over-limit request gets 200 with `limit` set to the clamped value, not 422. So the browse grid never trusts its requested page size; it uses the `limit` the response echoes back to compute `totalPages` and the pagination controls:

```tsx
const limit = data?.limit ?? 0;
const totalPages = limit > 0 ? Math.max(1, Math.ceil(total / limit)) : 1;
const currentPage = data?.page ?? page;
```

This is the right pattern for server-clamped pagination in this codebase: the response is authoritative.

When a fetch fails, the user can hit Retry - but if the failed fetch was for the current page, clicking Retry without changing `page` would be a React no-op (`useEffect` wouldn't re-run because the dependency array didn't change). The component uses a `reloadNonce` counter incremented on Retry, added to the effect's deps. The same-page refetch trick is small but worth surfacing: setting `page` to its current value would not re-run the effect.

The effect's cleanup function flips a `cancelled` flag that every async branch checks before calling `setState`. A page change mid-fetch can't leak state into the new page's render. Same pattern in `HealthBadge`. This is the React equivalent of an `AbortController` and is the central pattern for the client-side fetches in `BrowseGrid` and `HealthBadge`; adopting a query library was scoped out (§3 #3).

### 2v. Observability

The shopper exports its own metrics via `prom-client`, served at `/api/metrics`. The registry lives on `globalThis` (`app/lib/metrics.ts`):

```ts
declare global {
  var __avsa_metrics_registry: Registry | undefined;
}
export const registry: Registry =
  globalThis.__avsa_metrics_registry ?? (globalThis.__avsa_metrics_registry = buildRegistry());
```

This survives Next.js's dev-mode module hot reload - without `globalThis` pinning, each hot reload would build a fresh registry and the same metric name would be registered twice, throwing on the second call. The same trick is needed for any singleton on a Next.js server.

Two metrics are registered, both populated by the `/chat` route:

| Metric | Type | What it answers |
|---|---|---|
| `avsa_shopper_chat_requests_total{outcome}` | Counter | Per-outcome chat-proxy request count (`ok` / `upstream_error` / `bad_request`) |
| `avsa_shopper_chat_duration_seconds` | Histogram | Time from receiving `POST /chat` to streaming the first SSE byte back |

Plus Node-default process metrics under the `avsa_shopper_node_` prefix (heap, event loop lag, GC). The full observability inventory across services lives in `2.ARCHITECTURE.md` §2vi.

**`HealthBadge` as a live status indicator.** A small client component in the page header polls `/api/health` once on mount and displays one of three states (`pending`, `ok`, `unreachable`). It's deliberately not periodic - the badge answers `"can I reach the API right now?"` at page-load time, not `"is the API still up?"` over the session. A periodic re-probe would add a constant background fetch with little user-facing value; an explicit reload covers the operational case.

Verified: no `ErrorBoundary` component, no Next.js `error.tsx` route convention, no `componentDidCatch` anywhere in `app/`. A thrown render in `BrowseGrid`, `MessageStream`, or any other client component would white-screen the whole page rather than degrade gracefully to a fallback. The lift is small: an `app/error.tsx` per Next's convention catches everything below the layout boundary, and component-local boundaries can wrap specific subtrees that have known throw surfaces - the JSON-shape assumptions in `MessageStream` are the obvious candidate.

**No performance or accessibility CI gate today.** An earlier `lighthouserc.cjs` carried Lighthouse thresholds (perf ≥ 0.95, a11y = 1.0, best-practices ≥ 0.90, TTI < 3.5 s) but was never wired - no GitHub workflow invoked `@lhci/cli`, and the package wasn't in any `package.json`. The config was removed rather than left as a stranded artifact that implied a gate it didn't enforce. The intended posture remains worth keeping in mind for any future CI wiring: a perf/a11y budget that fails the build is a strict-enough discipline that thresholds need to be picked carefully against what the dev loop can actually sustain. A Lighthouse-CI step running `lhci autorun` against the standalone server, gated on changes to shopper source, is the smallest fix when the budget is decided.

### 2vi. Testing - unit, E2E, accessibility

Two layers under `frontend/apps/shopper/tests/`:

**Unit (`tests/unit/`, vitest + Testing Library + jsdom):** Nine specs covering:
- The page composition (`page.test.tsx`)
- Each component (`BrowseGrid.test.tsx`, `ChatInput.test.tsx`, `MessageStream.test.tsx`, `ProductCard.test.tsx`)
- The reusable library code (`consumeSse.test.ts`, `imageProxy.test.ts`, `next.config.test.ts`)
- The catalog BFF route (`catalog-route.test.ts`) - tests the proxy handler directly with mocked upstream

Each library module is testable in isolation - `consumeSse` accepts a `ReadableStream`, `imageProxy.proxyUrl` is a pure function. This is the central decision behind the lib/ split: the components are thin orchestration; the logic that can break is in pure functions.

**E2E (`tests/e2e/`, Playwright):** Ten specs that exercise the full chain (shopper → API → orchestrator → batcher → model → DB) against a stub-mode deployment:

| Spec | What it covers |
|---|---|
| `full-journey.spec.ts` | The end-to-end happy path: image upload → SSE stream → product cards rendered |
| `chat-flows.spec.ts` | Multi-turn submission, abort-on-resubmit, error rendering |
| `chat-image-upload.spec.ts` | Drag-drop, file picker, paste, size-cap rejection |
| `multi-turn-refinement.spec.ts` | The `X-Resume-Conversation-Id` lifecycle and `prior_result_ids` exclusion behaviour |
| `browse-grid.spec.ts` | Pagination, clamped-limit, retry |
| `images-proxy.spec.ts` | The signed-image proxy chain end-to-end |
| `health-badge.spec.ts` | The three states under live/dead upstream |
| `rate-limit.spec.ts` | The 429 surface from the API rate limiter |
| `navbar.spec.ts` | Conditional nav links (Grafana / API Docs) |
| `metrics-instrumentation.spec.ts` | `/api/metrics` returns the registered counters and histograms |

Two of the ten specs - `chat-image-upload.spec.ts` and `full-journey.spec.ts` - inject `@axe-core/playwright` directly and check accessibility at specific interactions (the upload tray, the rendered result stream). The other eight have no a11y gate today (§2v also names the absent perf/a11y CI gate). Adding axe to more specs is the natural lower-effort fix; a Lighthouse-CI build pass would be the higher-leverage one because it covers per-page state the interaction tests don't visit.

### 2vii. Build and deploy

`output: "standalone"` in `next.config.js` produces a self-contained Node server at `.next/standalone/apps/shopper/server.js`. The build is a single `pnpm --filter @avsa/shopper build` away; deployment is `node server.js`. No external Next.js runtime, no dependency on the source tree at run time.

`AVSA_API_URL` and `AVSA_MCP_URL` must be set in any non-local deployment. Both have localhost fallbacks (`http://localhost:8080`, `http://localhost:8082`) - unset in a deployed shopper, the runtime silently targets localhost rather than failing fast at startup. A boot-time check that flags an unset upstream when the runtime isn't on localhost would close that drift surface; it does not exist today.

Styling is BEM-style class names on top of a single `globals.css` (~870 lines) using CSS custom properties for design tokens. No CSS-in-JS, no Tailwind, no CSS Modules - the global file is auditable at this scope.

The Inter font is loaded via `next/font/google` with `display: "swap"` and the default Latin subset (`app/layout.tsx`) - avoids the flash-of-invisible-text that hurts FCP under cold cache. This is a perf hygiene choice that's independent of whether a perf CI gate is wired (§2v); it just makes the cold-cache first paint cheap.

The Docker image (`dockerfiles/Dockerfile.shopper`) is multi-stage pnpm/Next.js → `node-slim`, non-root. The Helm chart deploys it as `LoadBalancer` - the shopper is the only public LB in the cluster (see `2.ARCHITECTURE.md` §2iv.B).

Local dev is `pnpm dev` (or `just stack-up`, which starts the full stack including the shopper on `:3000`). The Playwright config can reuse an existing dev server or start its own.

## 3. Three architectural caveats

Three places where the frontend's current shape reflects a deliberate trade rather than a fully-productionised choice. Each is named with the trade-off taken, the current posture, and the resolution path.

**1. The page is `"use client"` from the top - no server-side data fetching for the initial render.** `app/layout.tsx` itself is a Server Component (default in the App Router); the client boundary lands at `app/page.tsx` and propagates from there. `page.tsx` opens with `"use client"` and uses `useState` for every piece of state on the page. The React tree is still server-rendered to HTML - Next.js's App Router sends the rendered shell to the browser and hydrates it client-side - but the data slots are empty because Client Components can't be `async` (React's render model doesn't support it). Data fetching has to land in `useEffect` after mount: `BrowseGrid` fetches the catalog there; `HealthBadge` polls there; `ChatInput` opens the SSE stream from the same place. The browser sees populated data only after those post-mount fetches return.

As a direct consequence, a browser with JavaScript disabled or blocked sees the layout chrome and nothing else - no catalog, no health badge, no chat. Moving `BrowseGrid` to a Server Component (the evolution below) would also restore at least the catalog view for that audience.

The trade-off taken is simplicity of mental model: the page is one component tree, all state lives in one place, no `"server"` / `"client"` boundary to reason about. This does mean that first-paint of populated content is slower than it could be (the catalog grid renders empty until the API responds), and there's no SEO-friendly initial HTML for the browse surface. The natural evolution if developing this beyond just a demo would be to convert the page to a Server Component, render `BrowseGrid` on the server with the initial catalog page baked into the HTML, and leave `ChatInput` + `MessageStream` as client islands. The component split already supports this - `BrowseGrid` could be rewritten as a Server Component fetching the catalog at request time, with no change to the `ProductCard` it renders or to how its pagination state flows.

**2. `<Image>` is `unoptimized` for catalog images.** `ProductCard` (`app/components/ProductCard.tsx`) renders catalog images with `unoptimized` set on the `<Image>` component, bypassing Next's image-optimization pipeline (no responsive resizing, no AVIF/WebP transcoding, no built-in CDN cache).

The reason is the signed-URL contract: catalog `image_url` is `/images/{key}?token=…&expires=…`, and Next's image-optimization cache keys by URL including query params. Two near-simultaneous requests for the same image with different tokens cache as two separate entries; when a token expires the cached optimized output is keyed against a URL that won't validate. Disabling optimization avoids these failure modes entirely and is the right choice given the current signing strategy.

The trade is real: the browser fetches full-resolution images even at small grid sizes, which on a multi-column browse grid is meaningfully wasteful bandwidth. Two evolution paths actually fix the cache mismatch:

(a) **Change the cache key off the URL** - sign by image-content hash so Next caches one optimized output per logical image regardless of the token. This is the larger lift. It needs a content-hash scheme computed at catalog-ingestion time and a hash → signed-URL lookup at request time on the API side; the catalog response then has to expose the hash to the client; and a Next.js custom image loader has to use all of the above.

(b) **Introduce a CDN layer** (Cloudflare Images, AWS CloudFront with Lambda@Edge) that owns both signing and transformation, so the shopper renders unsigned URLs and the CDN handles authorisation at fetch time. Option (b) is the cleaner long-term shape and the smaller integration surface - CDN configuration alone, no ingestion changes, no API-side hash lookup, no custom Next loader - though it requires CDN infrastructure that doesn't exist today.

Raising the signed-URL TTL would not fix this: Next's optimization cache is keyed on the full URL including the token.

**3. No client-state management library - pure `useState` for everything.** All state lives in component-local `useState`: the page holds `errorMessage`, `productCards`, and `sessionId`; `ChatInput` holds `text`, `items`, `isStreaming`; `BrowseGrid` holds `page`, `data`, `status`, `reloadNonce`. There is no Redux, no Zustand, no TanStack Query, no React Context for shared state. State that needs to flow between siblings (the session ID between `ChatInput` and the footer's Copy button) is lifted into the parent `page.tsx`.

This is defensible at the current scale: one user-facing page, five components, no global concerns (no auth state, no theme, no notifications queue, no shared cart). Pure `useState` is the simplest possible posture and avoids the framework lock-in a state library would impose. But it doesn't scale: a second user-facing route (`/products/{id}`, `/account`, anything) would either need its own copy of the session-id wiring or a shared mechanism, and either path becomes painful around the third route.

The natural evolution is staged: React Context for the session-id when a second route lands, and TanStack Query (or SWR) for shared data fetching the moment two components fetch the same upstream resource. Doing either today would be premature; doing both at the third route would be appropriate.
