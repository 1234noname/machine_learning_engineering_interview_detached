# API and Service Protocols

This document covers every wire surface between services in AVSA: the public HTTP/SSE edge the shopper hits, the Next.js BFF proxies that front it, the gRPC contract between the API service and the orchestrator, the asymmetric batcher contract, the model's image and text endpoints, and the cross-cutting concerns (signed image URLs, rate limiting, trace propagation). The MCP surface is its own deep dive in [4.MCP.md](./4.MCP.md); the QPS-tuning story for the batcher and model lives in [3.QPS.md](./3.QPS.md). Both are referenced where relevant rather than re-documented here.

Every behavioural claim is tied to a file or spec; known drift between the declared contract and the running code is named in §3 rather than papered over.

---

## 1. Protocol map

| From | To | Protocol | Spec / source |
|---|---|---|---|
| Browser | shopper (Next.js BFF) | HTTP + SSE | `frontend/apps/shopper/app/` |
| Shopper BFF | api | HTTP | [`specs/api/chat.openapi.yaml`](./specs/api/chat.openapi.yaml) |
| Shopper `/mcp` route | orchestrator MCP server | JSON-RPC 2.0 / Streamable HTTP | [`specs/mcp/tools.json`](./specs/mcp/tools.json) (see [4.MCP.md](./4.MCP.md)) |
| api | orchestrator | gRPC | [`specs/orchestrator/avsa.proto`](./specs/orchestrator/avsa.proto) |
| orchestrator | batcher | HTTP | `crates/batcher/src/routes/embed.rs` (singular per call) |
| batcher | model | HTTP | `apps/model/src/avsa_model/embed.py` (plural per batch) |
| orchestrator | model (text) | HTTP (direct, bypasses batcher) | `apps/model/src/avsa_model/embed_text.py` |
| api | Postgres (pgvector) | SQL | [`specs/db/`](./specs/db/) (see [DATABASE.md](./DATABASE.md)) |
| External MCP client | orchestrator MCP server | JSON-RPC 2.0 / Streamable HTTP | [`specs/mcp/tools.json`](./specs/mcp/tools.json) (see [4.MCP.md](./4.MCP.md)) |

Three contract languages cover the surface: OpenAPI for the public api (`/chat` is the load-bearing one), gRPC proto for the api → orchestrator hop, and JSON-RPC against the MCP manifest for the external tool surface. Internal HTTP between orchestrator → batcher → model is ad-hoc but small enough to inline in this doc. The browser only ever talks to the shopper BFF; the api is never directly reachable from outside the cluster.

The remainder of the doc covers each surface in turn (§2) and the known drift between declared contracts and the running code (§3).

## 2. Discussion

### 2i. Public API - the FastAPI edge

The API service mounts four routers (`apps/api/src/avsa_api/main.py`):

- **`GET /health`** - liveness only. Returns `{"status": "ok"}`. No DB or downstream probe.
- **`POST /chat`** - multipart upload + SSE streaming response. The conversational entry point.
- **`GET /catalog`** - paginated, stably-ordered browse.
- **`GET /images/{path}`** - HMAC-signed image proxy.

The single source of truth for the contract is [`specs/api/chat.openapi.yaml`](./specs/api/chat.openapi.yaml). Drift between the spec and the running service is intended to be caught in CI; one drift item survives today (see §3).

#### POST /chat - multipart upload, SSE response

The endpoint accepts `multipart/form-data` with a repeatable `image` field (up to `MAX_IMAGES = 8`, `apps/api/src/avsa_api/routes/chat.py:19`) and an optional `text` field (max 2,000 chars). Validation is layered:

1. **Rate limit** - `SlidingWindowLimiter.check/1` runs first on the client IP. A 60-second sliding window enforces `[api.rate_limit] requests_per_minute = 60` (config-driven, sourced from `config/avsa.base.toml`). Over-limit returns HTTP 429.
2. **At-least-one-input** - both `image` and `text` empty → 422.
3. **Image-count cap** - more than 8 images → 422 with a body naming the limit.
4. **Content-Length fast-reject** - declared size > `max_upload_bytes × image_count` (10 MiB × N) → 413, before any bytes are read.
5. **MIME allowlist** - `{image/jpeg, image/png, image/webp, image/heic}` → 415 on anything else.
6. **Streaming size gate** - `_read_limited/2` reads in 64 KiB chunks with a running total; over the cap mid-stream → 413. This is the authoritative size check because Content-Length doesn't cover chunked transfer encoding.
7. **Magic-byte verification** - `_valid_magic/2` checks the leading bytes against the declared MIME (`\xff\xd8\xff` for JPEG, `\x89PNG\r\n\x1a\n` for PNG, RIFF/WEBP, ftyp box for HEIC). Mismatch → 415.

Validated images plus the text are handed to `OrchestratorClient.stream_chat/3`, which opens a gRPC stream and yields card events. The response is `text/event-stream` framed as bare `data: <json>\n\n` lines - no SSE `event:`, `id:`, or `retry:` fields.

The OpenAPI declares `ChatEvent` as a discriminated union on `type`, but the union currently has exactly one variant: `product_card`. The discriminator structure is preserved because the AG-UI event cluster (`tool_start`, `tool_complete`, `intent_update`, `layout_hint`) was removed when the in-browser reasoning panel was cut - the scaffolding is retained so adding event types later doesn't break wire shape. Today's flow: zero or more `product_card` events, then the stream closes.

Verifier rejections (`:safety` and `:verification_failed`) and the LLM's `:no_tool_use` path do not emit a typed SSE error event - they emit zero `product_card`s and close the stream cleanly (`grpc_server.ex:98-118`). To the browser this is indistinguishable from "no matching products." The §3 entry names the resolution path; the caveat to surface here is just that an empty result list can mean "no matches" or "the verifier intervened" and callers cannot tell which.

Multi-turn resume is keyed off the `X-Resume-Conversation-Id` request header. Valid UUID → re-use as `conversation_id`; missing or malformed → server mints a fresh UUID. The chosen id is echoed back as `X-Conversation-Id` on the response. A client-supplied `X-Conversation-Id` is not honoured - that's a session-fixation guard. Inside the orchestrator the conversation_id is the lookup key for the per-session GenServer, and `prior_result_ids` from prior turns flow into the next kNN as an exclusion (see [4.MCP.md](./4.MCP.md) §2vii for the retrieval-side mechanism).

#### GET /catalog - paginated browse

Query: `page` (≥1) and `limit` (≥1; clamped to `[catalog] browse_max_limit`). The clamp is a soft cap - an over-limit request returns 200 with `limit` set to the configured ceiling, not 422. Page-past-the-end returns 200 with `items: []`. Malformed pagination (`< 1`) returns 422 (FastAPI's `Query(ge=1)`).

Results are ordered by a stable key (`id`) so paging is gap-free and dup-free across pages. Each item's `image_url` is signed at read time via `_sign_image_url/2`: the stored value is a tokenless `/images/{key}` path, and the response substitutes `/images/{key}?token=…&expires=…`. The catalog never returns a public-retailer URL - the signed proxy is the only image-fetch path callers see.

If the DB pool isn't configured (stub mode, missing `DATABASE_URL`), the route returns 503 with a structured error body rather than a 500.

#### GET /images/{path} - HMAC-signed proxy

The route owns the `/images/` URL shape; the underlying `StorageBackend` returns route-agnostic `SignedToken` objects. Outcome → status mapping:

| Status | Meaning |
|---|---|
| `200 image/*` | Valid token + object present (stored bytes streamed back) |
| `400` | `token` or `expires` query param missing - fail fast at the boundary |
| `403` | Token invalid, expired, tampered, or signed for a different path |
| `404 image/png` | Valid token but object absent - graceful PNG placeholder |

The 404-with-placeholder choice keeps the browse grid renderable when a catalog entry references a missing local file (a hermetic-fixture-vs-real-Fashion200k mismatch in dev); a deployed surface that needs hard 404s would adjust this route.

### 2ii. Shopper BFF - Next.js server routes

`frontend/apps/shopper/app/` carries five server-side routes that proxy the FastAPI surface. They exist for three reasons:

- **Single LoadBalancer:** The shopper's Service is the only public LB in the GKE deployment; the API service is ClusterIP behind the Ingress. Proxying through the shopper avoids cross-origin issues for the browser and keeps the API off the public internet.
- **Env-driven upstream:** `$AVSA_API_URL` and `$AVSA_MCP_URL` are read at request time, so local/staging/prod can repoint without a code change.
- **Forwarding-as-mechanism:** The proxies don't interpret payloads. The `/chat` route forwards the multipart body verbatim, propagates the resume header upstream, and streams the SSE response straight back. The `/mcp` route forwards the JSON-RPC body verbatim with `Authorization` preserved (see [4.MCP.md](./4.MCP.md) §2ii).

| Route | Upstream | Notes |
|---|---|---|
| `POST /chat` (`app/chat/route.ts`) | `$AVSA_API_URL/chat` | Multipart pass; forwards `X-Resume-Conversation-Id`; surfaces `X-Conversation-Id` back |
| `POST /mcp` (`app/mcp/route.ts`) | `$AVSA_MCP_URL` | JSON-RPC body verbatim; `Authorization` forwarded; see [4.MCP.md](./4.MCP.md) |
| `GET /api/catalog` (`app/api/catalog/route.ts`) | `$AVSA_API_URL/catalog` | Forwards `page`/`limit` only; 502 on upstream unreachable |
| `GET /api/health` (`app/api/health/route.ts`) | `$AVSA_API_URL/health` | `cache: no-store`; 502 on failure with `{status: "unreachable"}` |
| `GET /api/docs` (`app/api/docs/route.ts`) | `$AVSA_API_URL/docs` | Swagger UI proxy; pairs with `app/openapi.json/route.ts` so the docs UI's `/openapi.json` reference resolves |

Plus one Prometheus surface served by the shopper itself, not proxied:

- `GET /api/metrics` (`app/api/metrics/route.ts`) - Prometheus text exposition for `avsa_shopper_*` metrics (chat request count, chat duration histogram, Node default process metrics). Scraped by the local Prometheus job `avsa-shopper`.

### 2iii. Orchestrator gRPC - Conversation service

The proto (`specs/orchestrator/avsa.proto`) declares one service with two RPCs:

```protobuf
service Conversation {
  rpc StartConversation(StartConversationRequest) returns (ConversationEvent);
  rpc StreamConversationEvents(StartConversationRequest) returns (stream ConversationEvent);
}
```

`StartConversation` is unary - used by batch/non-streaming callers; returns only the last event emitted. `StreamConversationEvents` is server-streaming - what the API service uses, mapping each event to an SSE `data:` line as it arrives.

`StartConversationRequest` carries `string conversation_id` (client-supplied UUID, refused if two concurrent turns share an id), `repeated bytes image_bytes` (each capped at 10 MiB by the gateway before the RPC is invoked; multi-image is mean-pooled and L2-normalised inside the orchestrator), and `string user_text` (optional free-text prompt). The `repeated bytes` shape is a wire-compatible widening of an earlier singular `bytes image_bytes` field.

`ConversationEvent` is a oneof over `payload`. Today the oneof has exactly one variant: `product_result`. Field numbers 1 and 3–6 are explicitly reserved with their former names (`tool_call`, `ag_ui_tool_start`, `ag_ui_tool_complete`, `ag_ui_intent_update`, `ag_ui_layout_hint`) so a future message can never silently collide on the wire if the in-browser reasoning panel is revived. `ProductResultEvent` carries the catalog UUID, a similarity score in [0, 1], and a JSON-encoded `metadata_json` payload that matches the OpenAPI `ProductCard` schema verbatim - the schema is intentionally open at the catalog layer (arbitrary attribute keys) so the proto carries it as opaque JSON rather than a typed message.

**Buf governance:** (`specs/orchestrator/buf.yaml`) `lint: use: [DEFAULT]` is the baseline; five rules are explicitly carved out, each with a one-line justification in the file:

- `PACKAGE_DIRECTORY_MATCH` - buf expects `avsa/orchestrator/v1/` on disk; the proto lives at `specs/orchestrator/avsa.proto`.
- `SERVICE_SUFFIX` - buf wants `ConversationService`; the contract specifies `service Conversation`.
- `RPC_REQUEST_RESPONSE_UNIQUE`, `RPC_REQUEST_STANDARD_NAME`, `RPC_RESPONSE_STANDARD_NAME` - both RPCs share `StartConversationRequest` / `ConversationEvent` because the streaming and unary variants serve the same payload shape (one `ConversationEvent` oneof carries every event type).

Every other DEFAULT lint rule remains active and gates future proto changes.

Every gRPC call carries two headers built by `OrchestratorClient._build_grpc_metadata/1`: a W3C `baggage` header with `conversation_id=<uuid>`, and a simplified `traceparent` (`00-<trace_id>-<span_id>-01`) where `trace_id` is derived from `conversation_id` (hyphens stripped, zero-padded to 32 hex). This is deferred OTel: enough to correlate logs and metrics across the api → orchestrator hop today; the full OTel SDK integration with automatic instrumentation is named as a later-phase upgrade.

### 2iv. Batcher - the asymmetric contract

`crates/batcher/src/routes/embed.rs` exposes one POST endpoint and two GET endpoints. The interesting one is `/embed`, where the inbound and outbound contracts are deliberately asymmetric:

**Inbound (caller → batcher), singular per call:**

```json
{"image_bytes": "<base64-string>"}
```

**Outbound (batcher → model), plural per batched call:**

```json
{"images": ["<base64-string>", "<base64-string>", ...]}
```

The asymmetry is the throughput mechanism. The orchestrator issues N singular calls in flight; the batcher accumulates them up to `max_batch_size` or `max_wait_ms` (`config/avsa.base.toml [batcher]`) and emits one plural call to the model. The asymmetry is documented in [3.QPS.md](./3.QPS.md) §2i as the round-trip-amortisation lever - `num_drain_tasks=6` with `max_batch_size=24` is what produces the measured 157.3 QPS peak.

**Response (batcher → caller):**

```json
{
  "embedding": [/* f32 × 768 */],
  "attributes": {
    "category": "...", "colour": "...",
    "category_confidence": 0.0, "colour_confidence": 0.0
  }
}
```

The `attributes` field is additive: a caller that reads only `embedding` (an earlier version of the orchestrator did) is unaffected by the additive change. This is the backward-compat discipline applied consistently across both batcher↔model and batcher↔caller contracts.

**Error mapping** (`embed.rs:13`):

| Status | Condition |
|---|---|
| 400 | Base64 decode failure (client sent garbage) |
| 502 | `VitServiceUnavailable`, `MalformedVitResponse`, `ChannelClosed` - upstream fault |
| 504 | `Timeout` - upstream too slow; the timeout is `max_wait_ms × 10` |

The `MalformedVitResponse` includes mismatched-length checks on the model's parallel arrays (`embeddings.len() != attributes.len()`): if the model returns lists of unequal length, the per-image mapping is broken, and the batcher refuses to silently zip and ship corrupted data.

### 2v. Model - image and text endpoints

`apps/model/src/avsa_model/main.py` mounts three routers: `embed`, `embed_text`, `health`.

**`POST /embed`** - plural per call, matches `VitClient` in the batcher:

```
Request:  {"images": ["<base64-string>", ...]}
Response: {
    "embeddings": [[f32; 768], ...],
    "attributes": [
        {"category": str, "colour": str,
         "category_confidence": f32, "colour_confidence": f32},
        ...
    ]
}
```

`embeddings` is L2-normalised on the model side by convention (`vit.py:378`); the batcher independently re-normalises before returning to its caller (`batch_queue.rs:267`) - the batcher owns the unit-vector contract that pgvector's `<=>` cosine operator needs rather than relying on the model upholding it (a stub model, or a future un-normalised variant, would silently produce wrong cosine-distance results in pgvector without the batcher's re-norm). `attributes` is parallel (one entry per image) and additive (the batcher tolerates an absent `attributes` field for a stub model without heads).

**`POST /embed_text`** - plural, mirrors `/embed`:

```
Request:  {"texts": ["string", ...]}
Response: {"embeddings": [[f32; 512], ...]}
```

L2-normalised 512-d CLIP-text embeddings. The orchestrator calls this directly (bypassing the batcher) because text is small enough that batching wins nothing.

Error responses for both: `422` on empty input list (Pydantic validation + explicit guard), `400` on malformed payload.

**`GET /health`** - liveness; no model probe.

For CI and GPU-free local development, `AVSA_MODEL_STUB=1` swaps the ViT and CLIP encoders for deterministic SHA-256-based stand-ins that respect the wire shape (768-d and 512-d vectors, parallel `attributes` arrays). The full chat path runs in CI without any model weights, GPU, or network access - paired with the api's `AVSA_ORCHESTRATOR_STUB=1` (§2i) which short-circuits the orchestrator hop to a canned card, the chain is fully runnable for contract tests against the wire shapes.

Per-stage histograms inside `/embed` (`avsa_model_embed_decode_seconds`, `avsa_model_embed_embedder_seconds`, `avsa_model_embed_response_build_seconds`) split the handler time across base64 decode, the embedder forward, and response construction. These are the measurement substrate behind the QPS work documented in [3.QPS.md](./3.QPS.md) §2ii.

### 2vi. Cross-cutting concerns

The `StorageBackend.signed_url/2` interface (`libs/avsa-core/src/avsa_core/storage/__init__.py:50`) returns a `SignedToken { token, expires }` - the route layer (`apps/api/src/avsa_api/routes/catalog.py:_sign_image_url`) is what owns the URL shape (`?token=…&expires=…`). The local backend (`libs/avsa-core/src/avsa_core/storage/local.py`) uses HMAC-SHA256 over `path|expires` with the secret read from `AVSA_STORAGE_HMAC_SECRET` (env-driven, no fallback). Verification (`verify_signed_url/3`) recomputes the HMAC and compares in constant time via `hmac.compare_digest`. The secret never appears in catalog rows or in logs; rotation is by changing the env var (existing signed URLs become invalid at rotation time, by design).

`SlidingWindowLimiter` (`apps/api/src/avsa_api/middleware/rate_limit.py`) tracks per-IP request timestamps in a deque inside an `OrderedDict` (move-to-end on access, evict oldest when count exceeds `_MAX_TRACKED_IPS = 100_000`). Bounded memory under slow-drip multi-IP attacks. Rejections emit `avsa_api_rate_limit_total` (Prometheus counter). The `60 RPM` value is config-driven (`[api.rate_limit] requests_per_minute = 60`); the OpenAPI `x-rate-limit` annotation is illustrative, not authoritative - the config comment names this explicitly.

Two headers ride on every gRPC call from the API to the orchestrator: `baggage` (`conversation_id=<uuid>`) and a derived `traceparent`. This is enough to correlate logs and metrics across the hop today; full OTel SDK integration is deferred. The endpoint config (`config/avsa.base.toml [otel] exporter_endpoint = http://localhost:4317`) is in place for when the SDK lands.

`GET /health` is a process-liveness check: every service exposes one, returning `{status: "ok"}` or equivalent, with no DB or downstream probe. The readiness signal is the Prometheus `/metrics` endpoint, which on the orchestrator (`:9568`) answers 200 only after the DB pool, gRPC server, and MCP listener have registered with the supervision tree (per `application.ex`'s startup order - see [2.ARCHITECTURE.md](./2.ARCHITECTURE.md) §2iv.B). The CI E2E job exploits the difference: it polls `/metrics` because it needs the application ready, not just the process up. No service exposes a `/ready` distinct from `/health` - the metrics endpoint is the readiness boundary by convention.

### 2vii. Versioning and contract evolution

Three contracts, three approaches:

- **OpenAPI (`specs/api/chat.openapi.yaml`)** is the source of truth for the public API. The spec itself names CI-enforced drift detection ("drift between this file and a running service is a CI failure") but the surviving drift item - `GET /products/{id}` declared but not implemented - shows the check is incomplete in practice (see §3). Backward-compat strategy is additive responses: adding optional fields to existing schemas, never removing or renaming.
- **Proto (`specs/orchestrator/avsa.proto`)** is governed by buf with DEFAULT lint minus five scoped exceptions (§2iii). Field reservations (`reserved 1, 3, 4, 5, 6` for the removed AG-UI events) are the forward-compat discipline - old field numbers can never silently re-mean something new. `buf lint` enforces naming and structure on every change; **backward-compatibility detection is not configured today** (no `breaking:` section in `buf.yaml`, no `buf breaking` invocation in CI). The cleanest add is `buf breaking` against a baseline commit, gated on changes touching `specs/orchestrator/avsa.proto` in CI - would close the gap that today rests on review attention.
- **MCP manifest (`specs/mcp/tools.json`)** is the third contract - agent-facing rather than service-to-service or browser-to-edge. Conformance, the three known deviations from standard MCP, and the proposed `ajv-cli` CI guard all live in [4.MCP.md](./4.MCP.md) §3.

## 3. Known drift and boundaries

Three items where the declared contract and the running code differ, plus one deliberate design choice that callers should understand.

**`GET /products/{id}` is declared in OpenAPI but not implemented in the API service.** The spec describes it as a single-product lookup returning `ProductCard` (200) or `Error` (404); the running service has no `/products` router (`apps/api/src/avsa_api/routes/` carries only `health`, `chat`, `images`, `catalog`). The OpenAPI's own header claims drift is a CI failure - so either the CI check is scoped to `/chat` and `/catalog` only, or the endpoint was removed and the spec wasn't updated. The cleanest resolution is to remove the path from the spec until the endpoint is re-implemented; if external clients need single-product lookup, the implementation should land first and the spec entry should follow rather than predate it.

**The rate-limit annotation in the OpenAPI is illustrative, not authoritative.** OpenAPI `x-rate-limit: requests_per_minute: 30, burst: 10`; the running limit is `60 RPM` (no burst behaviour beyond the 60-second window) per `[api.rate_limit] requests_per_minute`. This is acknowledged in the config - the comment names the OpenAPI value as documentation only and points to itself as authoritative. The drift is intentional (the OpenAPI annotation is a hint of order-of-magnitude, not a contract), but a fresh reader of the spec gets a misleading number. The cleaner posture is either to align them (one source of truth) or to make the OpenAPI annotation explicitly non-normative with a description note.

**The SSE event taxonomy is single-variant; failure paths surface as empty result lists.** `ChatEvent` is a discriminated union with one variant (`product_card`). Verifier rejections (`:safety`, `:verification_failed`) and the LLM's `:no_tool_use` path produce zero events and close the stream cleanly (`grpc_server.ex:98-118`) - a client cannot distinguish "no matches" from "the verifier intervened" from "the LLM declined to call a tool." Exposing the verifier's reasoning as a structured error was avoided on purpose, but the indistinguishability is a real caveat for analytics, UX, and retry logic. The cleanest evolution is a typed `chat_error` event with a small reason taxonomy (`safety`, `verification`, `no_tool_use`); the existing discriminator structure accepts it additively without breaking existing consumers.

**OTel SDK integration is deferred; trace metadata is propagated manually.** The api → orchestrator gRPC call carries a `baggage` header with `conversation_id` and a simplified `traceparent` derived from the conversation UUID (zero-padded, fixed span_id). This is enough to correlate logs and metrics across the hop, but it isn't real W3C trace propagation: there's no span sampling, no parent-context lookup, no SDK-managed context propagation across async boundaries. The cleanest evolution is to wire the standard Python and Elixir OTel SDKs (the export endpoint config is already in place at `[otel] exporter_endpoint`) and remove the manual header builder. Until that lands, the manual scheme is what holds the trace story together - adequate for a system at this scale, not what a production observability story should look like at the next.

**The api → orchestrator gRPC channel is unauthenticated.** `OrchestratorClient` opens the channel with `grpc.aio.insecure_channel(self._addr)` - no TLS, no token, no mTLS. The trust boundary is the cluster-internal network: api and orchestrator both run as in-cluster pods with ClusterIP services, so traffic between them never reaches the cluster edge. Appropriate for a single-cluster deployment with no multi-tenant assumption; if the api ran outside the cluster (an external developer debug session, a second cluster) or the cluster grew multi-tenant, mTLS becomes load-bearing. The cleanest upgrade is GKE's workload identity + a service mesh sidecar (Istio or Anthos Service Mesh), which provides mTLS transparently: the channel constructor switches to `grpc.aio.secure_channel` with mesh-provided credentials, no proto changes needed.
