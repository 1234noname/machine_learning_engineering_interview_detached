# System Design - Service Breakdown and Deployment

AVSA is a small polyglot service mesh: each tier uses the language that fits its job, the services talk over typed contracts, and the whole thing runs three ways - natively for local development, on Kubernetes/GKE for a cluster deployment, and on Modal for serverless GPU. This document covers what each service is, how they communicate, how they're packaged, and how each environment deploys them.

---

## 1. Architecture and protocols

| Service | Language / framework | Role | Local port | In k8s |
|---|---|---|---|---|
| **shopper** | TypeScript / Next.js 15 | Web UI: image+text search, browse, SSE results; thin server-side proxies to the API and to the MCP server | 3000 | Deployment + LoadBalancer |
| **api** | Python / FastAPI | Public edge: `/chat` (SSE streaming), `/catalog`, signed `/images`, `/health`; sliding-window rate limit; gRPC client to orchestrator | 8080 | Deployment + ClusterIP (behind Ingress) |
| **orchestrator** | Elixir / OTP | Conversation brain: per-session GenServers, LLM planning (Claude), the image-native MCP tools, the five-check verifier (see [MCP.md](./MCP.md) §2vi); gRPC server + JSON-RPC MCP server | gRPC 50051, metrics 9568, MCP 8082 | Deployment + ClusterIP |
| **batcher** | Rust / Axum + Tokio | Throughput tier: accumulates singular `/embed` calls into batched model calls (`max_batch_size`, `max_wait_ms`, N drain tasks) | 8081 | Deployment + LoadBalancer |
| **model** | Python / FastAPI + PyTorch | Inference: ViT-base image embeddings (768-d) + attribute heads, CLIP text embeddings (512-d) | 8090 | Deployment + ClusterIP (GPU; disabled on GKE - served from Modal) |
| **Postgres + pgvector** | - | Catalog + image/text embeddings; cosine kNN over HNSW indexes; conversation state | 5434 | Cloud SQL / Neon |
| **Prometheus + Grafana** | - | Metrics scrape + dashboards (`avsa-baseline`) | 9090 / 3010 | ServiceMonitors per service + dashboard ConfigMap |

Per-service detail:

- **shopper** (`frontend/apps/shopper/`) - Next.js 15 app. Server routes proxy to the backend without CORS: `app/chat/route.ts` → API `/chat` (forwards multipart form + `X-Resume-Conversation-Id` for multi-turn resume; surfaces `X-Conversation-Id` back), and `app/mcp/route.ts` → orchestrator's JSON-RPC MCP server (`$AVSA_MCP_URL`, body verbatim, `Authorization` forwarded). Configured by `AVSA_API_URL` / `AVSA_MCP_URL`.
- **api** (`apps/api/src/avsa_api/main.py`) - mounts `health`, `chat`, `images`, `catalog` routers; no MCP routes (the MCP surface lives in the orchestrator - see [MCP.md](./4.MCP.md)). Talks to the **batcher** (`AVSA_BATCHER_URL`) and the **orchestrator** over gRPC (`AVSA_ORCHESTRATOR_ADDR`), and to Postgres for catalog and (signed) image delivery.
- **orchestrator** (`apps/orchestrator/apps/avsa/lib/avsa/application.ex`) - boots a gRPC server (`:50051`), a Prometheus metrics endpoint (`:9568`), and conditionally the MCP `Plug` server (`:8082`). Calls the **batcher** for image embeddings via `AVSA.EmbedStep` (which also surfaces the ViT attribute head output: `category`, `colour`, `category_confidence`, `colour_confidence`) and the **model** directly for text embeddings via `AVSA.TextTool` → `/embed_text` (bypassing the image batcher). Runs pgvector kNN through `AVSA.RetrievalTool` and calls Claude through `AVSA.LLM.Anthropic` (or a mock under `AVSA_LLM_STUB=1`). Three circuit breakers guard the downstream calls - `:batcher_circuit` (5 failures in 10 s, 60 s reset), `:anthropic_circuit` (3/10s/60s), `:text_encoder_circuit` (5/10s/60s) - installed at `application.ex:11-13`.
- **batcher** (`crates/batcher/src/lib.rs`) - Axum router exposing `/health`, `/metrics`, `/embed`. N drain tasks (`num_drain_tasks`) flush at `max_batch_size` or `max_wait_ms` (`config/avsa.base.toml [batcher]`); the upstream is `vit_service_url` - `localhost:8090` locally, the Modal endpoint in the cluster (via `helm/values.prod.yaml`).
- **model** (`apps/model/src/avsa_model/main.py`) - FastAPI app mounting `embed`, `embed_text`, `health` routers. Device via `AVSA_MODEL_DEVICE` (`cpu` / `mps` / `cuda`); `AVSA_MODEL_STUB=1` swaps a deterministic SHA-256 embedder for GPU-free CI. The ViT and CLIP encoders live in `vit.py` and `text_encoder.py`; the attribute heads in `heads.py`.

**Communication topology:**

```
Browser ──HTTP/SSE──▶ shopper (Next.js) ──┬─ /chat  ───▶ API (FastAPI)
                                          └─ /mcp   ───▶ Orchestrator MCP server (JSON-RPC)
                                                              ▲ (external MCP clients also enter here)
                          API ──gRPC StreamConversationEvents──▶ Orchestrator (Elixir)
                                                                   │  plan(LLM) → extract_attributes → find_similar → verify
                                                  image embed ◀────┤ (in-process AVSA.MCP.Tools)
                                                                   ▼
                                               Batcher (Rust) ──HTTP batched /embed──▶ Model (ViT + CLIP)
                                                                   │ text embed (direct ─ bypasses batcher) ▲
                                                                   ▼
                                                           Postgres + pgvector  (cosine kNN)
```

- **Browser ↔ shopper ↔ api** - HTTP + Server-Sent Events. The `/chat` turn streams product cards to the browser; the multipart upload accepts up to 8 images plus optional text (`MAX_IMAGES = 8` in `apps/api/src/avsa_api/routes/chat.py:19`), capped at `max_upload_bytes` per image (config-driven). Multi-turn resume is keyed off `X-Resume-Conversation-Id` (the client-supplied `X-Conversation-Id` is deliberately ignored as a session-fixation guard).
- **api ↔ orchestrator** - gRPC `StreamConversationEvents` (proto in `specs/orchestrator/avsa.proto`). The request carries `repeated bytes image_bytes` (multi-image; per-image embeddings mean-pooled then L2-normalised in the orchestrator) and `string user_text`. W3C trace metadata is propagated on the call.
- **MCP** - external clients hit the orchestrator's JSON-RPC server via the shopper's `/mcp` pass-through; the internal conversation flow reaches the same `AVSA.MCP.Tools` in-process. See [MCP.md](./4.MCP.md) for the full surface (one tool layer, two callers, two output shapes; image-native, not vector-native).
- **orchestrator → batcher → model** - HTTP. Image embeddings go through the batcher (for throughput); text embeddings go straight to the model (text is already short, batching wins nothing).
- **kNN** - pgvector `<=>` cosine over the 768-d image column and the 512-d text column; HNSW indexes on both.

Cross-cutting all of the above: every service exposes a Prometheus `/metrics` endpoint scraped into the central `avsa-baseline` dashboard, and the api → orchestrator hop additionally carries W3C `baggage` metadata so a single conversation can be followed across the chain. The metric inventory, tracing posture, and dashboard layout are covered in §2vi.

The remainder of the doc covers each of these areas in turn (§2) and the known drift between deployed and ideal (§3).

## 2. Discussion

### 2i. Configuration model

Configuration is layered, not flat:

- **Sources** (committed): `config/avsa.base.toml` carries the defaults across all environments; `config/avsa.{local,prod}.toml` are environment overlays.
- **Generated** (gitignored): `scripts/config-gen.py` merges base + overlay into `config/avsa.toml` - the runtime-effective file the services read. Invoked via `just config-gen <env>` (where `<env>` is `ci`, `local`, or `prod`).
- **Per-service loading**: Python and Rust services read the generated `config/avsa.toml` directly (`libs/avsa-core/src/avsa_core/config.py`'s `load_config()` is the canonical loader). The Elixir orchestrator additionally re-applies the profile overlay at boot via `AVSA.TomlConfig.apply_overlay/0` (`application.ex:9`, keyed off `AVSA_PROFILE`) - the same overlay logic, re-applied from source at process start so a redeploy can rebind without rebuilding `avsa.toml` upstream.

Environment variable overrides win over the file at runtime - `AVSA_BATCHER_URL`, `AVSA_MODEL_URL`, `AVSA_MODEL_DEVICE`, `DATABASE_URL`, `AVSA_MCP_API_KEY` are the load-bearing ones. Stub modes (`AVSA_MODEL_STUB`, `AVSA_LLM_STUB`, `AVSA_ORCHESTRATOR_STUB`) let most of the stack run without GPUs, LLM keys, or the Elixir service for CI and local work.

### 2ii. Data and migrations

Postgres 16 + **pgvector**; schema is versioned SQL under `infra/migrations/` applied by `migrate.sh`. The current ledger:

- `001_catalog_schema.sql` - products, categories, image refs
- `002_conversations_schema.sql` - per-conversation state
- (003 is absent from the ledger; the cause is documented in `006_drop_audit_schema.sql`'s preamble - 003 created the now-deprecated `audit` schema for LLM-spend tracking and gateway tool-call rows, removed when the MCP surface converged on the JSON-RPC server. The gap is preserved rather than renumbered; a future migration tool should accept gaps, and the cleanest fallback if it doesn't is a no-op `003_skipped.sql` placeholder, not a renumber that would invalidate external references to migration numbers. See [DATABASE.md](./7.DATABASE.md) §2iii for the full rationale.)
- `004_rename_price_column.sql` - small catalog rename
- `005_add_text_embedding.sql` - the dual-encoder rollout (image-only → image + text)
- `006_drop_audit_schema.sql` - the earlier audit log was removed when the MCP surface converged on the JSON-RPC server (audit lived in the deprecated HTTP gateway; see [MCP.md](./4.MCP.md) §3)

The catalog (`catalog.products`) carries a 768-d image column (`vector(768)`) and a 512-d text column (`vector(512)`) with HNSW cosine indexes on both. Conversation state is a separate `conversations` table. Seeding (`scripts/seed-catalog.py`) supports a hermetic `synthetic-v1` source for CI and the real Fashion200k source for quality measurement (the Fashion200k images and pre-computed embeddings live under `data/`, which is git-LFS).

### 2iii. Packaging (container images)

`dockerfiles/` builds one image per service; the multi-stage ones run as non-root:

| Dockerfile | Service | Notes |
|---|---|---|
| `Dockerfile.api` | api | Python 3.12-slim, `uv` install, serves on `:80` in-container (Helm exposes `:8080`) |
| `Dockerfile.model` | model | Python slim, base deps only (no torch in the cluster build - GPU model is served from Modal; this image is the CPU/stub fallback) |
| `Dockerfile.batcher` | batcher | multi-stage Rust → `debian-slim`, non-root |
| `Dockerfile.orchestrator` | orchestrator | multi-stage Elixir mix release → `debian-slim`, non-root |
| `Dockerfile.shopper` | shopper | multi-stage pnpm/Next.js → `node-slim`, non-root |
| `Dockerfile.loadtest` | loadtest | Locust harness (optional, enabled via `loadtest.enabled` in Helm) |

No GPU image is built today: the cluster has no GPU quota (see §3), so GPU inference is served from Modal and the cluster's `model` image is the CPU/stub variant. A GPU image and node-selector overlay are deferred until quota returns.

### 2iv. Deployment environments

#### A. Local development - `just stack-up`

Services run as native processes (uvicorn, the Rust release binary, `mix`, `pnpm dev`), not containers. Only stateful dependencies run in containers via docker-compose on Colima:

- Postgres 16 + pgvector - `infra/local-db/docker-compose.yml` (host `:5434`), migrated by `infra/migrations/migrate.sh` (`just db-migrate`).
- Prometheus + Grafana - `infra/local-observability/docker-compose.yml` (`:9090` / `:3010`).

`just stack-up` is the fast inner loop. The Justfile recipe sweeps the canonical service ports (8090, 8081, 9568, 8080, 3000) for listeners before starting - its own comment notes the sweep catches anything holding those ports, not just remnants of a previous `stack-up`, so an editor's dev server or a stray process is handled the same way. A container-based alternative exists: `skaffold dev` deploys the api/model/loadtest images to minikube via the Helm chart (port-forwards 8080/8000/8089). The previous `infra/e2e/docker-compose.yml` stack is no longer in the repo; end-to-end coverage runs through CI (`.github/workflows/ci.yml` `e2e` job).

#### B. Kubernetes / GKE - Helm

`helm/` is one chart deploying all five services as single-replica Deployments with matching Services. Layering:

- **Service exposure**: `batcher` and `shopper` are `LoadBalancer`; `api`, `model`, `orchestrator` are `ClusterIP`; `api` is reached through an Ingress (`helm/templates/ingress.yaml`, GCE class, routes `/` → `api-service`).
- **Config injection**: the batcher's `config/avsa.toml` is mounted via a ConfigMap (`helm/templates/batcher-config.yaml`); the prod overlay (`helm/values.prod.yaml`) sets `vitServiceUrl` to the Modal model endpoint, `maxBatchSize: 24`, and `numDrainTasks: 6` - the QPS-tuned configuration documented in [QPS.md](./3.QPS.md) §1. The in-cluster Rust batcher calls Modal for inference with the configuration that produces the measured 157.3 QPS peak.
- **Observability**: ServiceMonitors per service (`servicemonitor-{api,batcher,model,orchestrator}.yaml`) plus a Grafana dashboard ConfigMap (`grafana-dashboard.yaml`, enabled in the GKE overlay).
- **Values layering**: `values.yaml` (base) ← `values.local.yaml` (minikube: GPU 0, loadtest on) / `values.gke.yaml` (always-pull, monitoring + ingress on, GPU disabled - no T4 quota) / `values.prod.yaml` (digest-pinned `REGION-docker.pkg.dev/PROJECT/avsa/*` images, the direct Modal endpoint).
- **Build/deploy**: `skaffold.gke.yaml` builds via Google Cloud Build and `helm upgrade`s the `avsa` release with `values.gke.yaml`. Cluster, Cloud SQL, Artifact Registry, Secret Manager, IAM, and DNS are defined under `infra/terraform/` (modules + per-environment tfvars under `environments/{dev,staging,prod}/`).

#### C. Modal - serverless GPU

`modal_deploy/` hosts the GPU-bound tier. Three apps:

- **`model_app.py`** - the `AvsaModel` GPU class: `gpu="A10G"`, `min_containers=5`, `max_containers=6`, an `@modal.enter()` warmup that loads ViT + CLIP and runs dummy forwards (absorbing the `torch.compile` / cold-start stall). The class exposes two parallel HTTP surfaces:

  - **Direct `@modal.fastapi_endpoint` methods** - `embed_http`, `embed_text_http`, `healthz_http` - call `_do_embed` and `_do_embed_text` in-process on the GPU container, so the request path avoids the cloudpickle round-trip and Modal SDK cross-container dispatch that the shim path pays. This is the path the hosted solution uses (per the prod overlay referenced in §2iv.B): `https://erinversfeldcodes--avsa-model-avsamodel-embed-http.modal.run`.
  - **Legacy ASGI shim** (`model_api`) - a CPU FastAPI function that proxies to the GPU class via the Modal SDK. Retained for backward compatibility; **not** the deployed path. The shim's overhead - the same cloudpickle round-trip + cross-container dispatch the direct endpoints avoid - costs ~1.4 s per call at bs=24, and is the structural fix documented in [QPS.md](./3.QPS.md) §2i that took bs=24 service time from 2,001 ms → 831 ms and system throughput from 99 to 157.3 QPS.

  Weights are baked into the image at build time so cold starts only pay the load-into-VRAM cost (~10 s), not the ~600 MB download for ViT + CLIP.

- **`batcher_app.py`** - an async re-implementation of the batcher contract on Modal (looks up the model app via the Modal SDK). Available as an alternative deployment path; the deployed prod topology runs the Rust batcher on GKE and calls the Modal model endpoint directly.
- **`api_app.py`** - a serverless API shim (`avsa-api`) for standalone demos, running with a stubbed orchestrator (`AVSA_ORCHESTRATOR_STUB=1`) and managed Postgres (Neon) via the `avsa-api` Modal secret.

Deployed via `just modal-deploy-{model,batcher,api}` (or `modal-deploy-all`); wiring is by Modal secrets - each step's HTTPS URL is fed to the next. The rationale for choosing A10G over T4 / L40S / A100 / H100 lives in §2vi.

### 2v. Production topology

The deployed shape is split: the stateless services run on GKE, the GPU model runs on Modal.

```
                         GKE cluster (us-central1)
   Ingress ─▶ api ─gRPC─▶ orchestrator ─▶ batcher(LB) ──HTTP, ~831 ms service time at bs=24──▶  Modal A10G model
   shopper(LB)                │                                                                 (min 5, max 6 containers)
                              └─ pgvector kNN ─▶  Cloud SQL / Neon (Postgres)
                              └─ Claude API (Anthropic)
```

The single most important property of this topology: the GKE batcher → Modal model call crosses the public internet, and that round-trip is the binding throughput constraint at the current operating point. (Under different load shapes other constraints could become binding - LLM rate limits if `/chat` volume rose, kNN throughput if the catalog grew, Postgres connection pool under sustained turn rate - but at the system's current 157.3 QPS peak the cross-internet hop is what binds.) At the deployed batch size (24) the per-call service time is ~831 ms - pure network RTT is ~63 ms (TCP + TLS + propagation, per [QPS.md](./3.QPS.md) §2i Phase 2); the remainder is split across request serialisation, GPU-container request handling, and the model forward - not further decomposed in this work. The 831 ms is down from ~2,001 ms after the internal Modal shim was removed; [QPS.md](./3.QPS.md) §2i documents the structural fix. Every batching and scaling decision in QPS.md exists to amortise this per-call cost; it is the price paid for zero-ops GPU autoscaling, and the highest-leverage future change is collapsing the cross-internet hop (VPC peering, regional co-location, or - the larger lift - co-located GPU pods on the GKE cluster).

Expanding §1's per-service table along the deployment axis:

| Component | Local (`stack-up`) | GKE | Modal |
|---|---|---|---|
| model | native, CPU/MPS on `:8090` | (GPU disabled - served from Modal) | **A10G, 5–6 containers (direct fastapi_endpoint)** |
| batcher | native Rust `:8081` | pod, LoadBalancer → Modal | (async variant available; not deployed) |
| orchestrator | native Elixir | pod, ClusterIP | n/a (stubbed in `api_app`) |
| api | native `:8080` | pod behind Ingress | CPU shim (stub orchestrator) |
| shopper | `pnpm dev` `:3000` | pod, LoadBalancer | external |
| Postgres | docker-compose `:5434` | Cloud SQL | Neon (for `api_app` only) |
| observability | docker-compose Prom/Grafana | ServiceMonitors + Grafana ConfigMap | external |

Postgres differs across paths - Cloud SQL on the GKE deployment, Neon on the Modal `api_app` demo. They are not synchronised; a row written through the GKE path lives in Cloud SQL and is not visible from the Modal demo, and vice versa. Consolidation is deferred until the demo path is retired (or the GKE path is replicated on Modal as an alternative deployment).

### 2vi. Observability

The story §1 closes with - every service exports `/metrics`, the api → orchestrator hop carries trace metadata - has three layers under it, each with a current posture and a deferred upgrade.

Scrape topology varies by environment: docker-compose Prometheus + Grafana locally (§2iv.A), ServiceMonitors per service in GKE (§2iv.B). The single consolidated view is the `avsa-baseline` Grafana dashboard - defined under `helm/dashboards/` and mounted as a ConfigMap (`grafana-dashboard.yaml`) in the GKE overlay; locally Grafana runs on `:3010`. The choice to keep one cross-service dashboard rather than per-service dashboards reflects the operational reality: the load-bearing question for this system is *"which hop is slow?"* not *"which service is unhealthy?"*

A compact inventory of what each service exports - what's named in this table is the load-bearing signal, not the full set:

| Service | Metric | What it answers |
|---|---|---|
| api | `avsa_api_rate_limit_total` | Rejected request count (per-IP sliding window) |
| api | `avsa_api_orchestrator_call_duration_seconds` | End-to-end `stream_chat` call duration |
| batcher | `avsa_batcher_requests_total{outcome}` | Per-outcome request count (`ok` / `bad_request` / `timeout` / etc.) |
| batcher | `avsa_batcher_request_latency_seconds` | Per-request handler latency |
| model | `avsa_model_embed_decode_seconds` | Time in base64 decode inside `/embed` |
| model | `avsa_model_embed_embedder_seconds` | Time in the ViT forward |
| model | `avsa_model_embed_response_build_seconds` | Time constructing the response |
| orchestrator | `[:avsa, :tool_dispatch, :*]` | Per-tool dispatch counts |
| orchestrator | `[:avsa, :embed_cache, :hit/:miss]` | One-forward-per-turn observability with `modality` tag |
| orchestrator | `[:avsa, :attribute, :source]` | Per-attribute provenance: `vit` / `llm` / `text` |
| orchestrator | per-check verifier events | Boundary rejection distinguishable from response-side failure |
| shopper | `avsa_shopper_*` | Chat request count, chat duration histogram, Node process metrics |

Surface-specific context for each cluster is owned by the doc that owns the surface: the MCP-related events (tool dispatch, embed cache, verifier, attribute source) are at [MCP.md](./4.MCP.md) §2ix; the per-stage `/embed` histograms as QPS measurement substrate are at [QPS.md](./3.QPS.md) §2ii; the api edge metrics are at [5.API.md](./5.API.md) §2vi. This section is the index, not the duplicate.

The api → orchestrator gRPC call carries a W3C `baggage` header with `conversation_id` and a simplified `traceparent` derived from the conversation UUID (`OrchestratorClient._build_grpc_metadata`). Enough to correlate logs and metrics across that hop today; *not* real W3C distributed tracing - no span sampling, no parent-context lookup, no SDK-managed context propagation across async boundaries. The cleanest evolution is to wire the standard Python and Elixir OTel SDKs (the export endpoint config is in place at `[otel] exporter_endpoint = http://localhost:4317`) and remove the manual header builder; see [5.API.md](./5.API.md) §3 for the upgrade path.

Every service logs to stdout in a structured shape (`logging` in Python, `Logger` in Elixir, `tracing` in Rust). Locally `just stack-up` redirects each service's stdout to `/tmp/avsa-logs/<service>.log`; in GKE the default Cloud Logging integration captures stdout from each pod. No application-level log aggregator (Loki, Elasticsearch) is wired today - the deployment relies on Cloud Logging's built-in surface. The boundary-diagnosis log lines that matter (verifier rejection at `server.ex:158`, tool error at `server.ex:180`, circuit-breaker melts) are documented in the surface-specific docs.

`GET /health` on every service is a process-liveness check; the Prometheus `/metrics` endpoint is the readiness signal because it answers 200 only once the supervision tree has fully started (DB pool, gRPC server, MCP listener - per `application.ex`'s start order). The CI E2E job polls `/metrics` rather than `/health` for this reason - see [5.API.md](./5.API.md) §2vi for the distinction.

No formal SLI/SLO is set today. The 157.3 QPS peak ([QPS.md](./3.QPS.md) §1) and the MCP boundary screen rejection rate ([MCP.md](./4.MCP.md) §2ix) are the operational baselines a future definition would build on; the missing pieces are target latency percentiles (p50 / p95 / p99) on the chat surface and a per-error-class error budget. Defining these is the natural prelude to alerting - the `avsa-baseline` dashboard would gain threshold annotations and Grafana alert rules would emit pages on threshold violations.

### 2vii. Why this shape

- **Tool per tier:** Rust for the latency-sensitive batching loop - predictable concurrency, no GC pauses that could interfere with `max_wait_ms`. Elixir/OTP for stateful per-session orchestration and supervised crash isolation. Python for the ML stack and the edge API. TypeScript/Next.js for the UI with server-side proxies. Each boundary is a typed contract (gRPC proto, JSON-RPC against the MCP manifest, OpenAPI for `/chat`).
- **Throughput, brains, and edge separated:** The batcher isolates throughput tuning from model code and the orchestrator isolates conversation logic from the edge while the API remains the public boundary.
- **GPU on Modal, deliberately:** Serverless A10G gives zero-ops autoscaling and scale-to-zero out of hours, at the cost of the per-call service time that [QPS.md](./3.QPS.md) §2i quantifies. The trade-off is the most likely thing to revisit when usage moves the cost-vs-latency balance.
- **Why A10G, not T4 or H100:** Modal's GPU catalog spans T4 (~$0.59/hr) → A10G (~$1.10) → L40S (~$1.95, the mid-tier general-purpose card) → A100 (~$2.10–2.50) → H100 (~$3.95). The right choice depends on whether the GPU is the bottleneck - and it isn't: [QPS.md](./3.QPS.md) §2i shows the system is round-trip-bound, with GPU compute at ~9% of response time. Any tier upgrade attacks only that slice. A10G is the cheapest tier with adequate capacity (24 GB for ViT-base/16 at bs=24) and memory bandwidth (~600 GB/s), with mature Ampere Tensor Cores for fp16. T4 is ~46% cheaper but materially slower at fp16 - Turing-era Tensor Cores trail Ampere on this workload - a defensible cost-optimised fallback. L40S, A100, H100 cost 2–4× more for a slice the round-trip dominates - only justified once the round-trip is collapsed (a different deployment shape; see §2v).
- **Config-driven and stub-able (per §2i):** The `scripts/check-doc-drift.sh` step in CI (`.github/workflows/ci.yml:596`) closes the README ↔ Justfile loop: a setup command in the README that doesn't exist as a Justfile recipe fails the build.

## 3. Three caveats for a production deployment

Three places where this deployment differs from a fully-productionised shape. Each is named with the trade-off taken, the current posture, and the resolution path.

**1. GPU is not on the cluster.** The single most defining architectural choice in the deployed shape. The trade-off taken is serverless A10G - zero-ops autoscaling and scale-to-zero out of hours - at the cost of the ~831 ms per-call service time across the public internet that [QPS.md](./3.QPS.md) §2i quantifies. The current posture: `values.gke.yaml` disables GPU resources ("no T4 quota"); the GPU Dockerfile and T4 node-selector path are not in the repo. Every batching and scaling decision in QPS.md exists to amortise the round-trip - without the trade-off, that work wasn't needed.

The resolution path when quota returns is straightforward in shape but materially changes the system's binding constraint: a multi-stage `pytorch/cuda` Dockerfile, a `nodeSelector.model.cloud.google.com/gke-accelerator: nvidia-tesla-t4` overlay, and `resources.model.{requests,limits}.nvidia.com/gpu: "1"` flipped on in `values.gke.yaml`. The round-trip collapses to a sub-millisecond in-cluster call, and the QPS-tuned batching at `max_batch_size=24` / `num_drain_tasks=6` suddenly operates against a different binding constraint - most likely the GPU itself becomes the bottleneck, and the work shifts from "amortise the round-trip" to "tune the GPU forward." That's a different doc.

**2. Single replicas, no HA.** The trade-off taken is demo-cluster simplicity over availability. The current posture: every Helm Deployment is 1 replica; there is no `HorizontalPodAutoscaler`, no `PodDisruptionBudget`, no multi-AZ spread. Fine for a demonstration of the architecture; not suitable for tier-1 availability.

The resolution path is well-trodden for four of the five services but has a wrinkle for the orchestrator. For the stateless tier - `api`, `batcher`, `shopper` - the lift is `replicaCount: 3`, an HPA on CPU/memory thresholds, and a PDB with `minAvailable: 2`. For the orchestrator, the per-session GenServer state is the wrinkle: a turn arriving at a different pod than the one holding the conversation's GenServer needs either (a) sticky-session affinity on the Ingress so follow-up turns land on the same pod, or (b) a state-restore-from-DB story on pod startup - the orchestrator already writes turn records to Postgres via `AVSA.ConversationRecord`, so reading them back on session migration is the lift, not a redesign. Option (b) is the cleaner posture because it survives pod loss; option (a) is the cheaper posture because it works without writing recovery code.

**3. TLS not terminated.** The trade-off taken is demo simplicity. The current posture: `tls.enabled: false` in `helm/values.yaml`; the GCE ingress serves HTTP. The values stub is already in place (the file's own comment names `enabled: true`, `secretName`, and a `Certificate` resource as the three pieces needed).

The resolution path under GCE Ingress is a `ManagedCertificate` resource (GKE's built-in Let's Encrypt path) bound to the existing static IP. No third-party `cert-manager`, no manual rotation cycle, no certificate-renewal cron. Once the certificate provisions and `tls.enabled` flips, the GCE LB terminates HTTPS at the edge and the in-cluster traffic stays on HTTP - appropriate for a single-cluster deployment where the LB is the public boundary. End-to-end mTLS (HTTPS to the pods) is a larger lift and only relevant if the cluster grows multi-tenant or the threat model changes.
