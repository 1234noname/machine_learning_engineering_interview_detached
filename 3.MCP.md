# MCP in AVSA - Implementation and Surface

The Model Context Protocol (MCP) is the system's external tool surface. The same retrieval and attribute-extraction capabilities that drive the in-app shopper are exposed as agent-callable tools, so an external MCP client (the MCP Inspector, Claude Desktop, a future bespoke client) can drive the live catalog with the same fidelity as the first-party UI.

This document covers the surface as it actually exists in code, with every behavioural claim tied to a file and line, and known drift named in §3.

---

## 1. Architecture

```
                    external MCP client
                    (Inspector, Claude Desktop, ...)
                              │ JSON-RPC 2.0 / Streamable HTTP
                              ▼
            ┌─────────────────────────────────────┐
            │  shopper /mcp  (Next.js route)      │  pass-through, body verbatim,
            └─────────────────────────────────────┘  Authorization forwarded
                              │
                              ▼
       ┌──────────────────────────────────────────────┐
       │  AVSA.MCP.Server (Plug, port :mcp_port)      │  auth · body cap · screen_input
       └──────────────────────────────────────────────┘  initialize · tools/list · tools/call
                              │ dispatch
                              ▼
       ┌──────────────────────────────────────────────┐
       │  AVSA.MCP.Tools  (the L2 tool layer)         │  find_similar · extract_attributes
       └──────────────────────────────────────────────┘
                              │
       ┌───────────────┬──────┴──────┬───────────────────┐
       ▼               ▼             ▼                   ▼
  ImageResolver    EmbedCache   EmbedStep / TextTool   RetrievalTool
  b64 / ref → bytes  per-turn   ViT-768 / CLIP-text     pgvector kNN

  internal loopback (request):  AVSA.Conversation → AVSA.MCP.Tools  (in-process, no socket, no boundary screen)
  external return path: card maps (canonical UUID strings, catalog metadata) - wrapped by to_card/1
  internal return path: %AVSA.ProductResult{} structs (raw 16-byte UUIDs)   - via find_similar_results/2
```

The MCP surface lives entirely in the Elixir orchestrator. The Python API service exposes no MCP routes - an earlier HTTP gateway (`/mcp/tools`, `/mcp/tools/call`) was removed when the surface converged on the JSON-RPC server. The Next.js `/mcp` route is a pure pass-through and never interprets the JSON-RPC payload.

## 2. Discussion

### 2i. One tool layer, two callers, two output shapes

The core design is a single L2 tool module - `AVSA.MCP.Tools` (`apps/orchestrator/apps/avsa/lib/avsa/mcp/tools.ex`) - reached two ways:

- **Internal loopback**: `AVSA.Conversation` (the per-session GenServer driving each chat turn) calls `AVSA.MCP.Tools.extract_attributes/2` and `AVSA.MCP.Tools.find_similar_results/2` directly (`conversation.ex:129-137`). No socket, no JSON serialisation, no input screening - that protection lives on the output side of the turn instead (§2v).

- **External HTTP**: `AVSA.MCP.Server` (`apps/orchestrator/apps/avsa/lib/avsa/mcp/server.ex`) is a JSON-RPC 2.0 Plug. After auth and input screening, it dispatches the same underlying functions for outside clients (`server.ex:208-227`).

The two callers get two output shapes from the same retrieval path (shown in the §1 diagram captions). The mechanism is constrained by JSON encoding: in-process consumers - `AVSA.Verifier.check_factuality`, `AVSA.GrpcServer.build_product_event/1` - pattern-match on `%AVSA.ProductResult{}` and use the raw 16-byte binary UUID directly. That binary is not valid UTF-8 and crashes `Jason.encode!`, so the external surface wraps results in card maps via `to_card/1` (`tools.ex:237-251`) which canonicalise the UUID to a 36-char string and drop the struct nesting. The cards are deliberately not round-trippable back to the struct.

The shared retrieval path is the real design win; the wire surface still requires explicit per-tool registration. Adding a third tool means three required registrations - a new function in `tools.ex`, a new `call_tool` clause in `server.ex` (the dispatch is hardcoded - `defp call_tool("find_similar", ...)`, `defp call_tool("extract_attributes", ...)`, `defp call_tool(_, _) → :unknown_tool`), and a new entry in `specs/mcp/tools.json` - plus optional wiring into `conversation.ex` if the in-app turn flow should drive it. The "one tool layer" claim holds at the path level, not the registration level.

### 2ii. A single conformant MCP server

`AVSA.MCP.Server` implements MCP `2025-03` over Streamable HTTP + JSON-RPC 2.0. The three methods a standard client expects are all dispatched in `dispatch/3` (`server.ex:132-165`):

- `initialize` - returns static `protocolVersion = "2025-03"` (`server.ex:80`), `capabilities = {tools: {}}`, and `serverInfo = {name: "avsa-mcp", version: "1.0.0"}`. The version is pinned; multi-version negotiation is deferred - see §3.
- `tools/list` - reads `specs/mcp/tools.json` from disk and returns its `tools` array. The path is configurable via `:avsa, :specs_root` so tests can point at fixtures.
- `tools/call` - runs `screen_arguments/1`, then `dispatch_tool/3`, which mints a fresh `request_id` of the form `"mcp-#{UUID}"`, runs the tool, and purges the embed cache for that id in an `after` clause (`server.ex:209-227`).

Errors use the standard JSON-RPC codes - `-32700` parse, `-32600` invalid request, `-32601` method-not-found, `-32602` invalid-params, `-32603` internal - plus two AVSA-specific extensions: `-32001` unauthorized and `-32010` payload-too-large (`server.ex:70-76`).

The Next.js `app/mcp/route.ts` is a thin `POST /mcp` pass-through to `$AVSA_MCP_URL`. It reads the body as text, forwards `Authorization` when present, sets `content-type: application/json` upstream, and streams `upstream.body` back unchanged. It does not parse the JSON-RPC payload - method routing happens server-side. The proxy exists so the shopper LoadBalancer fronts the MCP surface without cross-origin issues or exposing the orchestrator's MCP port directly. Off-the-shelf MCP clients connect without an adapter; the protocol surface is small enough to audit by reading `server.ex` end to end.

### 2iii. Image-native tools, not vector-native

Both tools take an image (`image_b64` inline base64, `image_ref` storage key, or `image_b64_list` for multi-image) or text, and embed internally. There is no `embedding` argument anywhere (`tools.ex:7-37`).

`find_similar/2` is modality-aware (`tools.ex:216-227`):
- with image arguments → embed to 768-d ViT, run image kNN via `AVSA.RetrievalTool.call/2`
- with `text` → embed to 512-d CLIP-text via `AVSA.TextTool`, run text kNN via `AVSA.RetrievalTool.call_text/2`
- multiple images via `image_b64_list` are mean-pooled into one query vector and re-L2-normalised before the kNN (`embed_and_pool/2`, `mean_pool/1`, `l2_normalise/1`)

`AVSA.MCP.ImageResolver` (`image_resolver.ex`) turns the image argument into bytes: `image_b64` decoded inline with zero I/O, or `image_ref` read from the local storage root via `default_local_resolver/1`. The ref is treated as opaque; `Path.expand` resolution is bounded to a `safe_root <> "/"` prefix check, so any `..`-style traversal is rejected with `{:error, :image_ref_traversal}` (`image_resolver.ex:104-117`) - the protection is mechanism-based, so it holds against unknown future refs as well as known ones.

An external MCP client cannot produce a 768-d AVSA-ViT vector - it doesn't have the ViT weights. A tool that demanded a pre-computed embedding wouldn't actually be agent-callable. Making the tools image-native is what makes them usable; the acceptance bar stated in the server module docstring is concrete - "Inspector calls `find_similar` WITH AN IMAGE → grounded catalog results."

### 2iv. The per-turn invariant and how it's measured

`find_similar` and `extract_attributes` are usually both called on the same image in a single turn. Both route through `AVSA.EmbedCache.with_embed/5` (`embed_cache.ex:75-97`), keyed by `(request_id, modality, :crypto.hash(:sha256, image_bytes))`, so the expensive ViT forward executes once per turn per distinct image. Two image-consuming tools touching the same image incur one forward; two touching different images (a comparison tool, or a follow-up uploaded by the user) incur two - the cache scopes by content hash, not by request.

Errors are not cached (`embed_cache.ex:91`) so a transient failure does not poison the turn; the next tool call retries. Entries have a default 60 s TTL (`embed_cache.ex:50`) reaped lazily on next lookup (`embed_cache.ex:117-131`), so a long-running turn could in principle re-use an entry up to a minute old before the TTL elapses, after which the next lookup re-runs the forward.

Two cleanup mechanisms keep the cache bounded:

- Externally, every MCP `tools/call` mints its own `request_id` of the form `"mcp-#{UUID}"`. `call_tool/2` wraps the dispatch in `try / after AVSA.EmbedCache.purge_request(request_id)`, so the cache entry is gone before the JSON-RPC response goes out (`server.ex:209-227`).
- Internally, `AVSA.Conversation` mints a plain `Ecto.UUID.generate()` (`conversation.ex:75, 93`) and `AVSA.GrpcServer` purges at turn teardown after the verifier has run.

Because telemetry and logs carry the `request_id`, the `"mcp-"` prefix separates MCP-originated cache scopes from chat-originated ones for dashboard filtering. The cache lives below the MCP boundary and never leaks across calls.

Whether the invariant holds in the wild is observable from telemetry: `with_embed/5` emits `[:avsa, :embed_cache, :hit]` and `[:avsa, :embed_cache, :miss]` events on every probe with a `%{modality: modality}` tag (`embed_cache.ex:83, 87`). A turn that touched the same image with both tools should show one miss + one hit in the Grafana panel; deviations are diagnosable from the metric without instrumenting calls.

### 2v. Trust posture - boundary screening and the verifier

`AVSA.MCP.Server` treats the external HTTP boundary as untrusted. `screen_arguments/1` runs the text arguments - `text`, `user_text`, `image_description` - through `AVSA.Verifier.screen_input/1` **before** any tool dispatches (`server.ex:149-161, 187-204`). A failure returns a JSON-RPC `invalid_params` (`-32602`) error, not a tool result, so an untrusted caller cannot reach the LLM (`extract_attributes`) or the catalog (`find_similar`) with an injection or unsafe payload.

The loopback path is intentionally not screened on input. The in-app `AVSA.Conversation` flow proposes a response, and `AVSA.Verifier.check/2` (the full pass) gates on the output side (`conversation.ex:154-178`). The internal text has already been through that gate before any tool result reaches the user - input-screening the loopback would be duplicate work; not screening the external surface would leave a hole. The trust boundary is in one place: untrusted input is screened where it enters, trusted input gets the heavier output-side check.

`AVSA.Verifier.check/2` runs five gates in sequence (`verifier.ex:111-117`), short-circuiting on the first failure:

1. **catalog_resolvability** - every product UUID in the proposed response resolves in the catalog (no hallucinated IDs)
2. **pii_filter** - regex-driven PII detection on the assistant text
3. **factuality** - claimed attributes match what the catalog stores for the cited products
4. **injection_pattern** - regex-driven prompt-injection sniff on the assistant text
5. **safety** - regex-driven safety probe on the assistant text

`AVSA.Verifier.screen_input/1` runs the boundary-relevant subset of these - `:injection_pattern` followed by `:safety` (`verifier.ex:99-100`). The catalog, PII, and factuality checks are response-side concerns and don't apply to incoming text arguments, so screening at the boundary uses only what's meaningful pre-dispatch.

The conversation flow's response-side handling retries once on a non-safety verifier failure: it re-asks the LLM for a plan with the same input and re-runs the verifier on the new response (`conversation.ex:158-178`). A `:safety` failure terminates immediately; a second verifier failure returns `:verification_failed`.

### 2vi. Operational guarantees

- Bearer token via `Plug.Crypto.secure_compare/2` (constant-time), config-driven via `:avsa, :mcp_api_key` (`server.ex:42-52, 234-266`). When unset, the server is open - convenient for a local Inspector demo. When set, every request must carry `Authorization: Bearer <key>`; a missing or wrong token is rejected with HTTP 401 + JSON-RPC `-32001`.

- Because both tools accept an inline base64 image, `call/2` reads the body with a hard `:length` cap (`:avsa, :mcp_max_body_bytes`, default 16 MiB - room for a ~12 MB source image as base64 plus the JSON-RPC envelope). `Plug.Conn.read_body/2` returns `{:more, _, conn}` on overflow, which is rejected as HTTP 413 + JSON-RPC `-32010` rather than buffered (`server.ex:88-108`). The cap is total request bytes, not per-image: for an n-image `image_b64_list` request the per-image budget falls to roughly 12/n MB of source bytes - two ~6 MB source images fit, two ~8 MB do not. The cap was sized for the single-image case the in-app shopper drives. If multi-image becomes a load-bearing external use case, the cleanest evolution is a per-image (or per-image-count) cap with the total still bounded - rather than a flat lift of the total, which would relax the protection the cap exists to provide.

- `AVSA.Application.mcp_children/0` (`application.ex:92-119`) refuses to start an exposed server (`MIX_ENV=:prod` or `AVSA_MCP_EXPOSED=1`) with no `:mcp_api_key` set. The error message names the misconfiguration explicitly. A misconfigured deployment never lands as an unauthenticated public tool endpoint - it fails fast at boot.

- No rate limiting lives at this Plug: a valid bearer token can issue requests as fast as the listener will accept them. If per-key or per-IP throttling becomes a requirement, it would sit in front of (or be added to) the server module - the boundary is the natural place for it, alongside auth and the body cap.

Exposure is a conscious, gated decision, and the one image-accepting endpoint cannot be used to push an unbounded payload at the listener.

### 2vii. The tool surface - composability and the manifest

The manifest (`specs/mcp/tools.json`) defines two tools with full `inputSchema` / `outputSchema` and MCP `annotations`:

- **`find_similar`** - top-k catalog retrieval; pure vector search, no LLM (`"makesSdkCall": false`, `"readOnlyHint": true`). Optional `attrs` (with an explicit `colour`) narrow the candidate set before ranking.
- **`extract_attributes`** - returns the four-key attribute object (`category`, `colour`, `formality`, `occasion`) that feeds `find_similar`. LLM-backed (`"makesSdkCall": true`).

`find_similar`'s cross-turn behaviour is the most interesting internal: the conversation flow threads `prior_result_ids` into the attrs map between turns (`conversation.ex:139-142`), and the retrieval tool applies this list as a **SQL exclusion** on the kNN - past results are filtered *out* of the next response (`retrieval_tool.ex:183, 247`) - so a follow-up turn shows new candidates rather than re-surfacing what the user already saw. This is an internal-loopback concern and is not part of the public MCP tool surface; external MCP callers manage their own conversational state.

`extract_attributes` composes three sources of attribute information when an image is supplied (`attribute_tool.ex`):

1. **ViT attribute head** supplies `category` and `colour` (the image-derived attrs).
2. **The LLM** is constrained to supply only `formality` and `occasion` - the mechanism is a narrowed tool schema (`attribute_tool.ex:184-200`): when `vit_attributes` is non-nil, the LLM is handed an `extract_attributes` tool whose `input_schema` declares only `formality` and `occasion` as properties, with the explicit description "Do NOT infer category or colour - those are supplied by the image attribute head." The narrow schema is the enforcement; `compose_attrs/2` (`attribute_tool.ex:122-133`) then merges by taking `formality`/`occasion` from the LLM response and `category`/`colour` from `vit_attributes`.
3. **Text-keyword override** runs last (`attribute_tool.ex:141-156, 164-167`): the user text is scanned against a hardcoded `@colour_vocab` (~17 colours) and `@category_vocab` (~16 categories); a match overrides whatever ViT supplied for that attribute. So if the user uploads a red dress photo and types "show me the navy version", the composed `colour` is `"navy"`, not the ViT-inferred `"red"`.

Per-attribute source provenance is emitted as telemetry: `emit_source/2` (`attribute_tool.ex:169-175`) fires `[:avsa, :attribute, :source]` with `%{attribute, source}` metadata tagging each composed attribute as `"vit"`, `"llm"`, or `"text"`. Dashboards can break down attribute distribution by source - useful for catching cases where text-keyword override dominates (a sign the ViT head is weak on the relevant category) or where the LLM is filling in for missing ViT output (a sign the image is out of distribution). When no image is supplied, the LLM extracts all four attrs against the wider tool schema (`attribute_tool.ex:202-217`) and text override still runs.

The split is deliberately composable: extract structured attributes once, then retrieve (and re-retrieve with tweaked attributes) cheaply. The `makesSdkCall` annotation lets a caller reason about which tools incur LLM cost.

### 2viii. Tests and CI integration

Two further layers of telemetry beyond the cache events of §2iv and the per-attribute source provenance documented in §2vii:

- **Tool dispatch** - `[:avsa, :tool_dispatch, :find_similar]` and `[:avsa, :tool_dispatch, :extract_attributes]` events with `%{count: 1}` measurement (`tools.ex:87, 138`). These answer *"is the tool dispatching"* - the metadata is empty by design (no latency, no payload size, no failure-code dimension). Operational depth for per-call performance lives in the underlying L1 paths (the batcher and model histograms covered in [QPS.md](./2.QPS.md)), not at the MCP boundary.
- **Verifier** - per-check telemetry via `emit_telemetry/3` (`verifier.ex:139`), so a boundary `:injection_pattern` rejection is distinguishable from a response-side `:factuality` failure.

The operator-visible log signal sits at the server boundary: `Logger.warning` on input-screening rejection (`server.ex:158`) carries the tool name and which check rejected; `Logger.error` on tool execution failure (`server.ex:180`) carries the tool name and inspect-formatted reason. The ephemeral telemetry counters are the dashboard signal; the logs are the per-incident trail.

CI integration is concrete (`.github/workflows/ci.yml`): the orchestrator unit-test job runs `mix test` with `:start_mcp` set to `false` (`config/test.exs:15`), so no listener boots and tests exercise the server module directly via `Plug.Test`. The E2E job brings the orchestrator up with `AVSA_MCP_API_KEY` set, waits on the Prometheus `/metrics` endpoint to confirm boot, runs Python E2E tests against the MCP surface (`AVSA_E2E_MCP_KEY` shared with the listener key), and then runs Playwright tests against the shopper's `/mcp` proxy. The test directory `apps/orchestrator/apps/avsa/test/avsa/mcp/` carries three suites - server (auth, body cap, dispatch, screening), tools (modality, output shape, embed-cache scoping), image resolver (b64 decode, traversal rejection).

## 3. Deviations from standard MCP

Three places where this implementation departs from a strict reading of MCP `2025-03`. Each is named with the spec position, the implementation position, and the resolution path.

**1. The `find_similar` response shape does not match its declared `outputSchema`.** The manifest declares `results[]` as `{result_id, score, rank}` with `additionalProperties: false` and `rank` required (`specs/mcp/tools.json`); `to_card/1` returns `{result_id, score, title, category, price_cents, image_url}` (`tools.ex:237-251`) - no `rank` field, four fields the schema rejects. A schema-validating MCP client would refuse the response. This is the most consequential of the three: it is a real interop failure for any strict client.

The preferred resolution is to **widen the schema to match the cards**. External clients building product UIs need `title`, `image_url`, `category`, and `price_cents` to render results; trimming the cards to the manifest's minimal shape would defeat the use case the tools were designed for. `rank` can be derived from array position by callers that need ordinal position; if a manifest-declared `rank` is genuinely needed, add it to the card map rather than removing the catalog fields. The cleanest CI guard against recurrence is an `ajv-cli` validation step against a representative `tools/call` response, gated on the orchestrator-test job.

**2. Protocol-version negotiation is not implemented.** The server pins `protocolVersion = "2025-03"` as a static string in `initialize` (`server.ex:80`) and returns it regardless of what the client requested. MCP allows a server to declare a single supported version, so this is conformant for the current revision - but a client that has moved to a future version will see a server claiming the previous one, with no clear failure mode at the protocol layer.

The cleanest evolution when MCP next moves: read the client's `protocolVersion` from the `initialize` request, match it against a small declared set the server supports, return the negotiated version if there is a match, and reject unknown versions with `-32602` (`invalid_params`) plus a clear error message. Implementing this today is deliberately deferred - the deployed surface targets one MCP revision - but the gap will bite the first time a multi-version client connects.

**3. The manifest carries a non-standard `_comment` top-level field.** `specs/mcp/tools.json` opens with a `_comment` describing the rationale for an `annotations` extension inside `tools[]` items. The annotation is useful in-place documentation, but `_comment` would not be a known field to a strict MCP validator; the manifest could be rejected before any tools are listed.

The annotation belongs somewhere a validator won't see it: a sibling `specs/mcp/README.md` carrying the same prose, or a JSON Schema `$comment` field inside the `tools[]` items themselves (which is a known JSON Schema keyword and is ignored by validators by design). Either resolves the conformance gap without losing the information. Of the three deviations in this section, this is the most cosmetic - and the easiest to fix.
