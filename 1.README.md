# AVSA - Agentic Visual Shopping Assistant

A conversational, image- and text-driven product search system over the [Fashion200k](https://github.com/xthan/fashion-200k) catalog. You upload a photo (or describe what you want), and the system extracts attributes, runs vector similarity search, verifies the results, and streams product matches back to a web UI - and exposes the same capabilities as MCP tools for external agents.

> **On scope.** The original brief (see [`0.ASSIGNMENT.md`](./0.ASSIGNMENT.md)) is a narrow ML-serving task: maximise the QPS of a ViT image classifier. For a full-stack ML role I wanted to send signal across a broader set of engineering attributes, not pure-ML ones, and to make MCP a first-class part of the design rather than an afterthought. So I built AVSA: a working multi-modal search demo, with system and model QPS measured against committed baselines, grounded in real Fashion200k data. I've also used it to show a few patterns I keep in my back pocket, which means the design is heavier than the current scale justifies but ready to absorb a much larger scope without a rewrite.

Briefly, AVSA allows one to browse from amongst 5000 fashion images and use a combination of images and text to filter and find what one is looking for. The hosted demo can be found [here](http://34.171.72.185/) and metrics from your interactions will show up in the linked Grafana dashboard. Please note that there is a fixed, demo-sized query budget attached to the hosted app so chances are if it stops working then that budget has been spent :)

To run this project yourself:
```
just setup # will pull in data/ from git LFS
export AVSA_STORAGE_HMAC_SECRET=$(python3 -c 'import secrets; print(secrets.token_hex(32))')
export AVSA_ANTHROPIC_API_KEY=<your Anthropic key>
just stack-up
```

If you don't have a personal AVSA_ANTHROPIC_API_KEY, please run as follows instead:
```
just setup # will pull in data/ from git LFS
export AVSA_STORAGE_HMAC_SECRET=$(python3 -c 'import secrets; print(secrets.token_hex(32))')
AVSA_LLM_STUB=1 just stack-up
```

To bench system and model QPS locally yourself with the committed config, please run:
```
# System QPS
just bench-qps

# Raw model QPS
AVSA_BENCH_PROD_TARGET=model AVSA_PROD_MODEL_URL=http://localhost:8090 just bench-qps
```

And to see the MCP demo run:
```
just stack-up
just mcp-demo
```

To tear down the stack, run `just stack-down`.

## Deviations from the brief

| Brief | This build | Why |
|---|---|---|
| ViT as a 1000-class **ImageNet classifier** | Same ViT backbone (`vit-base-patch16-224`) repurposed as a **768-d embedding** model + two lightweight **attribute heads** (category, colour) | The real task is product *retrieval*, not classification |
| `api` emits **synthetic** images | A real **Fashion200k catalog** with browse + a signed image proxy; synthetic fixtures kept only for tests | Needed a realistic retrieval corpus |
| "Maximize QPS" by tuning the Python model | QPS preserved, but met by a dedicated **Rust batching tier** (dynamic micro-batching, `max_batch_size=8` / `max_wait_ms=50`) in front of the model | Out-of-process batching amortises GPU kernel launches without touching model code |
| QPS is the single metric | QPS is **one gate among several**: retrieval recall@k, attribute accuracy, and end-to-end p95 latency | Quality matters once it's a real product |
| Pure Python | **Polyglot**: Python (model/api), Rust (batcher), Elixir/OTP (orchestrator), TypeScript (frontend) | Right tool per tier |
| `skaffold dev` on minikube | Local dev via **`just stack-up`** - services run as native processes, with docker-compose Postgres/Grafana on Colima; skaffold retained for minikube/GKE deploy, **Modal** added for serverless A10G GPU | Faster inner loop; containers reserved for stateful deps and cloud |

## Scope expansions

**MCP (the focus).**
- Two tools exposed over MCP - `find_similar` (image- or text-native kNN; no LLM) and `extract_attributes` (LLM-backed) - defined in [`specs/mcp/tools.json`](./specs/mcp/tools.json).
- **Image-native tool design:** tools accept raw images/text and embed internally; callers never pass AVSA-specific vectors.
- A Python HTTP **MCP gateway** (`/mcp/tools`, `/mcp/tools/call`; API-key auth, audit log, session push into the live SSE stream) plus an Elixir **JSON-RPC 2.0 / Streamable-HTTP MCP server** (MCP `2025-03`: `initialize`, `tools/list`, `tools/call`) with **input screening at the external trust boundary**.
- A Next.js **MCP client proxy** route in the frontend.
- **Dual tool manifests** (Anthropic `specs/llm/tools.json` vs MCP `specs/mcp/tools.json`) kept in sync by a **drift check in CI**.

**Conversational orchestration.** An Elixir/OTP **orchestrator** (gRPC, per-session `Conversation` GenServers) plans each turn with an LLM (Claude), calls the tools, and caches the image embedding **once per turn** so two tools touching the same image trigger a single ViT forward.

**Safety / verification.** A six-check verifier (catalog resolvability, PII, factuality, prompt-injection, tool-schema sandbox, safety) gates every response before it reaches the user.

**Full-stack frontend.** A Next.js 15 **shopper** app: image/text search, paginated catalog browse, and SSE-streamed results.

**Retrieval platform.** Postgres 16 + **pgvector** (HNSW) for cosine kNN over dual **image (768-d ViT) and text (512-d CLIP)** embeddings.

**Production concerns.** Rate limiting, HMAC-signed image URLs, circuit breakers, an audit trail, Prometheus/Grafana observability, and fully config-driven tuning.

## Architecture at a glance

```
Browser (Next.js shopper) ──HTTP/SSE──▶ API (FastAPI)
                                          │  gRPC                 ┌─ MCP gateway (/mcp/tools/*)
                                          ▼                       └─ external agents
                               Orchestrator (Elixir/OTP)
                            plan → extract_attributes → find_similar → verify
                                   │ embed (per-turn cache)        │ kNN
                                   ▼                               ▼
                              Batcher (Rust)               Postgres + pgvector
                                   │ batched /embed
                                   ▼
                              Model (ViT + CLIP, Modal GPU)
```

## Notes

- `data/` (Fashion200k images, precomputed embeddings, attribute heads) is git-LFS.
- Stub modes (`AVSA_*_STUB=1`) let most of the stack run without GPUs, a DB, or LLM keys for CI and local dev.
