# Maximising Model QPS - Final Configurations and Reasoning

We measured two distinct throughput numbers to answer two different questions:

**Model QPS** is the GPU's (or CPU's) intrinsic ceiling - embeddings per second produced by the forward pass in isolation, with no batcher, no network, no HTTP. It is set by the model's computational structure and the hardware's ability to execute it: precision, batch-size effects on Tensor-Core tile filling, and memory bandwidth.

**System QPS** is the user-facing throughput: client → batcher → model → response. It is bounded by the model ceiling but also by everything that surrounds it - the round-trip cost per call, batching efficiency, in-flight concurrency, container parallelism, framing overhead at HTTP boundaries, and the behaviour of the client model itself. In the deployed shape, system QPS can be described as:

```
System QPS  ≈  (in-flight batches) × (mean batch size) / (per-call service time)
```

- though only two of those terms are directly configurable: `num_drain_tasks` sets the in-flight batch count, and `max_batch_size`/`max_wait_ms` set the upper bound and timing window on the batcher's mean batch size. The mean batch size *achieved* at steady state is an outcome, determined by client behaviour and arrival rate against those settings - which becomes load-bearing later. The optimisation problem is to identify which of the three terms is binding, configure the controllable bounds wide enough not to clip it, and then improve the term that remains.

Throughput is determined by how the forward executes - kernel choice, precision, fusion, batching, concurrency - not by the numeric values in the weights themselves. Optimising QPS therefore means optimising execution, not retraining. Every adopted change in this work was gated on an equivalence check (cosine ≥ 0.9999 vs the fp32 reference embeddings) and an attribute-accuracy bound: speed wins that fail the quality gate were considered negative results, not wins.

The remainder of the document covers: the final configurations reached (§1); the reasoning paths on GPU (§2i) and on local hardware (§2ii); the measurement discipline and quality gate that backed every adopted change (§3); and the exploration paths whose answers would most change the operating picture if the work continues (§4).

## 1. Final configurations

### GPU - Modal A10G

| Component | Setting |
|---|---|
| Precision / runtime | `fp32 + torch.compile(reduce-overhead)` |
| Batcher | `max_batch_size = 24`, `num_drain_tasks = 6` |
| Modal | `min_containers = 5`, `max_containers = 6`, HTTP endpoint placed directly on the GPU class (`@modal.fastapi_endpoint` - the second Modal hop has been removed) |
| Per-call service time | bs=24 = **831 ms** (from 2,001 ms before the hop removal; controlled probe, n=10 - see §4 on probe-vs-live difference) |
| **GPU ceiling** (in-memory, fp16 + compile) | bs=8 **425 QPS**, bs=24 **348 QPS** |
| **System QPS peak** | **157.3 QPS** at c=512 (measured from an in-region pod) |
| Operating cost | ~$4,000/month standing for the 5 always-warm A10G containers; ~$0.012 per 1k embeddings at sustained peak |

### CPU/MPS - Apple Silicon

| Component | Setting |
|---|---|
| Device / precision | MPS fp16 |
| Server | single uvicorn worker, no async dispatch (per measurement - see §2ii) |
| Batcher | `num_drain_tasks = 1`, `max_batch_size = 24` (cap is non-binding under the deployed client model) |
| Preprocess | `torchvision.transforms.v2` (replacing the HF feature-extractor) |
| **Embedder in-process** (model QPS proxy) | bs=8 **130 QPS**, bs=24 **135 QPS** |
| **System QPS** | **61.2 QPS** (rigorous N=3 + 1 warmup, per-run range ≤ 2%) |

The two configurations differ in every dimension, which is itself the most important high-level finding: the constraints on the two surfaces are structurally different, so the optimisation paths and the achievable ceilings have nothing in common despite running the same model.

## 2. Discussion

### 2i. GPU - the hosted path

The system improved from a 33.3 system QPS baseline to **157.3 QPS** - a 4.7× gain - and **425 model QPS**. The work split, conceptually, into two phases that compounded: three structural levers that took it to 99 QPS, then a single architectural fix that took it from 99 to 157.3. The model itself contributed nothing measurable at the system level; the reasoning below explains why that was predictable from first principles, and why measurement still mattered to confirm it. (The phases below describe the order in which each part of the work was understood and de-risked; in practice some lever measurements - early fp16 in particular - ran in parallel with the structural sweeps.)

The hosted model runs on a serverless GPU endpoint called from a batcher in a different cluster, over the public internet. A serverless GPU call carries a round-trip cost in the hundreds of milliseconds; a ViT-base forward at bs=24 on A10G fp16 takes ~74 ms (in-memory measurement). The binding cost on the system surface was therefore almost certainly the round-trip, not the GPU. The work was sequenced on that basis: amortise the round-trip first, characterise the GPU ceiling, then test only the model-side levers that could plausibly beat the deployed baseline.

**Phase one - amortising the round-trip:** The Rust batcher's job is to absorb single-image client requests and emit fewer, larger model calls. Three knobs control how well it does so, and each had a quantitative hypothesis attached:

- **`max_batch_size` 8 → 24** (+81%). At a multi-hundred-ms cost paid per call, the cheapest amortisation was more images per call. The optimum tracked `concurrency / containers`: at c=64 with 3 containers, bs=24 (2.7 batches in flight) kept all three busy; bs=32 (2.0 exactly) idled one. The sweep landed bs=24 as the point of best utilisation.

- **Batcher drain-task concurrency 1 → 6** (+197%, the largest pre-collapse lever). If batcher was restricted to running a single drain task it would holding one batch in flight regardless of how many GPU containers running. The throughput ceiling was therefore `max_batch_size / RTT` independent of container count. Running N drain tasks allows N batches to overlap. A sweep over N ∈ {1, 3, 5, 6, 7} placed the optimum at N=6 with `min_containers=5`, where concurrent calls match the warm pool without queue starvation (N=7 regressed: queue arrival rate could not supply that many full batches).

- **Container scaling 3 → 5** (+18%). More containers fill 24-image batches more consistently. Sub-linear because the round-trip is paid per call regardless of count.

These three levers, compounded, lifted the system from 33.3 to 99 QPS - accounting for essentially all of the gain before any model-side work was attempted.

**Phase two - instrumenting the round-trip and removing what it found.** With structural levers exhausted at 99 QPS, the next move was to measure what the round-trip actually was. A probe run from inside the deployment cluster decomposed it layer by layer. Raw network was small: TCP+TLS ~63 ms total, one-way propagation ~21 ms. The bulk of the ~2 s bs=24 service time lived inside a second Modal hop: the hosted call path was two-hop - batcher → HTTP → a Modal CPU shim container → Modal SDK RPC → GPU class. The internal `.remote.aio()` RPC carried **~1,395 ms at bs=24** (a cloudpickle round-trip plus a cross-container dispatch).

Placing a FastAPI endpoint decorator directly on the GPU class makes the GPU container serve HTTP itself, with the inference function called in-process - no CPU intermediary, no second serialisation, no second control-plane crossing. Two implementation details mattered: the Modal decorator had been renamed (`web_endpoint` → `fastapi_endpoint`), and FastAPI's per-request worker threads break `torch.compile(reduce-overhead)` CUDA graphs because they tie to the original capture thread (fixed by pinning all CUDA work to a single-thread executor). With those in place, the bs=24 service time dropped from **2,001 ms to 831 ms (−58%)**; system throughput reached **157.3 QPS** measured from an in-region pod.

**Phase three - characterising the GPU ceiling.** The deployed baseline is `fp16 + torch.compile(reduce-overhead)`. CUDA graphs collapse the forward into a single `cudaGraphLaunch` with zero per-kernel dispatch overhead - which is exactly what the most-common GPU optimisation techniques (graph fusion, dispatch elimination, tactic search) target. So the question was not "what GPU lever exists" but rather what could plausibly beat this baseline at the hosted batch size. Two categories were worth measuring:

- *Precision and runtime variants within fp16:* bf16 (Ampere has separate bf16 Tensor Cores), `torch.compile` max-autotune (Triton tile search beyond reduce-overhead's defaults), Flash Attention 2 (tiled attention to cut HBM traffic), ONNX Runtime with the CUDA execution provider, TensorRT FP16 with dynamic batch profiles, and torch_tensorrt with static engines. Each had a specific reason to be tested, tied to a hypothesis about where the baseline might leave headroom. Every one regressed at bs=24 against the deployed baseline, each for a reason tied to the CUDA-graph + fp16 starting point already being strong. The investment in this group was bounded by construction: each test was a single-file config flip or small code change deployed against the already-warm pool, with marginal cost equal to a two-pass warm bench run; none required calibration data, weight conversion, or new GPU provisioning. Compared to the calibrated INT8 work below - which required an offline calibration dataset and engine compilation before any bench - these were stand-up costs only.

- *Calibrated INT8 on the A10G's INT8 Tensor Cores.* This is structurally different from the fp16-based levers and would require investment to explore fully: a calibrated static INT8 model runs real INT8×INT8→INT32 matmuls on Tensor Cores, theoretically ~4× fp16 GEMM throughput. It is the lever that, if anything, would shift the GPU ceiling.

The INT8 commitment was deliberately staged to avoid burning expensive GPU time on faith. The decisive cheap experiment is a batch-size sweep: the signature of a compute-bound INT8 win is advantage growing monotonically with batch, because larger batches make the GEMMs more compute-bound and amortise the fixed quantise/dequantise conversion overhead. That sweep ran at bs ∈ {8, 24, 64, 128, 256} via ONNX Q/DQ format + TensorRT execution provider: tied at bs=8 and bs=256, +9% at bs=64, −13% at bs=128. The signature was absent, and both INT8 and fp16+compile sat flat at ~310–400 QPS across bs=24–256.

The most likely root cause (inferred from the consistent failure of every weight-precision lever against the activation-volume scaling) is that at bs ≥ 24 the A10G is activation-memory-bandwidth-bound, not GEMM-compute-bound (activations alone reach ~300 MB at bs=256). The levers in this set reduce weight traffic and the activation traffic, which dominates at these batch sizes, is invariant under weight precision. Direct profiling via nsight or DCGM counters would confirm; the indirect evidence from the sweep is sufficient to act on. The fp16 + CUDA-graph baseline is the end of the road for this model on this GPU. Calibrated INT8 code was reverted, with the diagnostic batch-size sweep preserved as the durable answer to "should we have done INT8."

**What would actually move the GPU number next:** inside Modal's serverless topology there are no further structural levers worth pulling - the internal hop is gone, the GPU compute path is exhausted at the activation-bandwidth limit, and additional containers do not reduce the per-call round-trip. The only remaining lever that would materially move the number is collapsing the round-trip itself: co-locating the GPU on the same cluster as the client (a GPU node pool on GKE), for a sub-millisecond in-cluster call. That is a longer-horizon trade - Modal's zero-ops autoscaling against dedicated GPU-node management - and it is the kind of decision that arrives when the serverless platform stops being a fit for steady-state load, not from optimisation pressure alone. With the round-trip collapsed, the activation-bandwidth ceiling would bind on a different surface, model compute would start mattering, and the GPU-side levers ruled out above would deserve re-measurement against a new baseline.

One caveat that shapes the system's operating profile rather than its peak is that the benchmark uses synchronous, blocking clients - each issues one request, blocks on the response, then issues the next. Local measurement under this model showed mean batch size pinned at ~8 regardless of total concurrency, because at moderate arrival rates the queue cannot supply more before `max_wait_ms` fires. At the prod peak operating point the arithmetic indicates the opposite: 157.3 QPS at 6 drain tasks and 0.831 s service time implies a mean batch of ~22, near the 24-image cap - so at c=512, batches are filling close to `max_batch_size`. The synchronous-client constraint therefore binds at intermediate concurrency, not at peak. What it shapes is the latency profile (every user waits a full round-trip before issuing the next request, so per-request response time grows with c) and the throughput achievable at moderate load. A pipelined client model - multiple in-flight requests per user - would primarily move the latency picture: lower per-request response time at intermediate operating points, and a usable throughput closer to peak at lower c. It is the cleanest still-untaken measurement, and would shift how the system is operated as much as how it performs.

### 2ii. CPU/MPS - the local path

The local stack ended at **61.2 system QPS** and **130 model QPS**, with three structural Apple-Silicon limits diagnosed and three classes of "more concurrency" lever rejected with mechanism. The single kept win was a preprocess change worth +5.4%. The reasoning path matters more than the number, because what it surfaces is that the local platform's binding constraints are structural and not removable above the model.

The expected first lever was going to be moving the forward to the GPU. Profiling on CPU/fp32 showed the forward at ~90% of embed time; the obvious win was to run it on the Apple GPU via MPS. `AVSA_MODEL_DEVICE=mps` lifted raw throughput +80% (56 → 100 img/s). Adopting this surfaced a one-line bug that had silently re-stomped the resolved device to CPU on every Mac, masking the win until removed. The tell was a device-vs-CPU comparison returning exactly cosine = 1.000000 (both runs were the same device, embeddings bit-identical).

fp16 on MPS added a further +9% over MPS fp32. bf16 was probed directly and matched fp16 exactly - Apple GPU shader cores use a single 16-bit compute path with no separate bf16 hardware (unlike CUDA Ampere, where bf16 Tensor Cores are physically distinct). `torch.compile` regressed in every mode tested: MPS has no CUDA SMs, so Triton is unavailable and the compile path falls back to a slower implementation while still paying the compilation overhead.

Earlier preliminary numbers (single-run, no warmup) had been carrying ~30–40% run-to-run variance - the same bs=24 baseline read 99.6 one day, 57.8 another, with an intermediate cold-start at 28.8. In part this was because these experiments were run on a personal laptop where other services competed for resources. To compensate, the measurement harness was rebuilt to use N=3 measured runs + 1 warmup pass, median reported, per-run range recorded. Under that discipline, the local stack delivers **61.2  system QPS** through the full HTTP path; the embedder itself delivers **130 model QPS** at bs=8–24. The 2.3× gap between the two surfaces was the question to answer.

The natural first hypothesis was the FastAPI handler: a synchronous embed call inside an `async def` route blocks the event loop, so concurrent HTTP requests should serialise rather than overlap. Wrapping the embed call in `anyio.to_thread.run_sync` is the textbook fix. Measured: **−6 to −11% at every concurrency level**, well outside the noise floor. The mechanism is Apple-Silicon-specific: PyTorch's MPS backend holds the GIL through GPU dispatch (CUDA releases it for the GPU work), so the thread pool buys no parallelism and pays per-call dispatch overhead.

Two further concurrency probes followed from a different angle: if the FastAPI event loop wasn't the bottleneck, perhaps batcher-side concurrency or model-process parallelism was. `batcher.num_drain_tasks = 3` (the lever that had been the largest hosted-solution win) and `uvicorn --workers 2` (separate model processes each serving their own queue) both regressed by the same ~10%, for the same root cause: **MPS serialises GPU dispatch across threads, processes, and contexts**. No software lever above the model produces parallel GPU work on Apple Silicon. This is also the cleanest cross-surface check on the principle that local is not predictive of prod - the same drain-task lever that failed on MPS (−10%) was the largest prod lever (+197% on CUDA), because the GIL behaviour differs at the dispatch boundary.

Once concurrency was ruled out, the `/embed` handler was instrumented per stage. Framing overhead (base64 decode, Pydantic validation, JSON encode) measured at ~3 ms total - not the ~25 ms originally hypothesised. The remaining 22 ms gap between in-process embedder time (~60 ms at bs=8) and the same call inside the HTTP server (~82 ms) lived **inside the embedder call itself** when there was an idle gap before it. A controlled probe - same code, clean process, 55 ms inter-call sleep - reproduced the slowdown: total embed time +14 ms, with both the GPU forward (+7 ms) and the CPU extractor (+5 ms) getting slower. The mechanism is Apple SoC power management: 50 ms idle (approximately the batcher's `max_wait_ms`) is enough to drop the GPU clock and evict CPU caches, and the next call pays the recovery cost. A GPU keep-alive ping recovered ~5 ms in isolation but only ~1.5 ms in the real model server (where async + Prometheus + HTTP overhead added cost the keep-alive could not reach). Identified, real, but not productively fixable from inside the model server.

Replacing the HF `ViTFeatureExtractor` with `torchvision.transforms.v2` removed CPU↔MPS hops in the preprocess pipeline (the HF path forced CPU work between GPU stages): **+5.4% system QPS** (58.0 → 61.2). The equivalence cosine was 0.999767 minimum on 32 real images (0.999951 mean) - the worst-case image sits 1.3×10⁻⁴ below the 0.9999 gate. The change was kept with that caveat noted; recall@5 re-verification is a tooling follow-up.

Profiling after the change showed the forward is still 89–91% of in-process embed time on MPS fp16. The earlier plan to chase preprocess once the forward shrank past an Amdahl threshold (25–30%) was based on the wrong premise: MPS+fp16 shrank the forward, but everything else shrank proportionally. JPEG-decode swaps were retired without further measurement - Pillow 11.3 in the venv is already compiled against libjpeg-turbo (so the proposed "swap to libjpeg-turbo" was a no-op), and the theoretical maximum win from `torchvision.io.decode_jpeg` was ~1% of system time.

Every local measurement ran on a personal laptop - browser, sync daemons, Spotlight indexing, occasional video playback - under macOS's aggressive thermal and power management. The N=3 + warmup protocol (§3) bounds within-session variance to ≤ 2% on steady-state points, but cross-session drift driven by external state (other processes contending for CPU and the unified memory bus, GPU clock state, thermal headroom at the moment the bench starts) is wider than on prod, where containers are isolated and the fleet is homogeneous. The structural findings above - MPS dispatch serialisation across threads/processes/contexts, power-cycling cost after idle, the in-process-vs-HTTP gap closing to ~3 ms of framing overhead - are robust to that drift because each reproduced across multiple sessions and has a mechanism that explains why it reproduces. The exact numerical headlines (61.2 QPS system, 130–138 QPS in-process, +5.4% from the preprocess swap, +13.6 ms per call after 55 ms idle) carry a few percent of irreducible surface uncertainty on top of the within-session range. This is a structural limit of the dev-box measurement environment, not a measurement-discipline gap, and is part of why the local surface is best used to surface and explain mechanisms rather than to quote exact comparative numbers.

Inside the existing stack, the levers have largely been exhausted; the three structural MPS limits (dispatch serialisation across threads/processes/contexts, aggressive power management adding idle-recovery cost, and the synchronous-client feedback loop that pins steady-state batch size) cannot be worked around in software above the model. The realistic paths forward are platform changes, not optimisations. Native Apple frameworks (MLX, or Core ML via coremltools) bypass PyTorch's MPS backend and have more direct access to Apple's compute fabric; published transformer benchmarks suggest 1.5–3× over PyTorch MPS for inference-shaped workloads. Switching the dev box to a small CUDA machine would eliminate every MPS-specific limit in one move and bring the local stack into closer parity with hosted behaviour. The Apple-native route is the higher local ceiling at a larger engineering commitment; the CUDA-dev-box route is the cheapest path to local-prod parity. Choosing between them is a workflow decision, not a performance one, and was scoped out of this pass.

## 3. Methodology and quality

Three things made the results above trustworthy enough to act on.

Both surfaces use the same harness (Locust → batcher → model HTTP path for system QPS; an in-process embedder probe for model QPS) with config-driven parameters (`n_runs`, `warmup_passes`, `concurrency_levels`, `batch_sizes`, `spawn_rate`, `run_duration_s`). The harness was custom-built rather than reused from a standard load-testing template, because the relevant question - *how does this specific batcher amortise this specific round-trip under this specific client model* - required the load generator to encode the deployed client wire contract (singular `{"image_bytes": …}` requests to the batcher, plural `{"images": […]}` to the model) and to report metrics at both surfaces simultaneously. A general-purpose load test could measure throughput at one endpoint but would not have surfaced the in-process-vs-HTTP gap that turned out to be the central diagnostic on the local surface.

Each `(batch_size × concurrency)` point runs for a fixed 30–60 s window after one warmup pass that is discarded. The default is N=3 measured runs per point; the median is reported and the per-run range is recorded. Per-run range observed in practice is ≤ 2% on warm steady-state points, and the warmup pass is load-bearing - without it the first measured pass reads 30–40% high. Comparisons made in this work are between points captured under this discipline; earlier preliminary numbers that used a single un-warmed pass have been retracted where they conflict. One scope note: this discipline bounds within-session variance only. On the local surface, where the host is shared with the developer's normal workload, cross-session drift remains wider than the within-session range - see §2ii ("A limit on what the local numbers can claim") for the caveat this places on specific local headlines.

All precision and runtime levers were run through an equivalence check (cosine similarity vs the fp32 reference, gated at 0.9999 minimum across the ablation image set) and an attribute-classification accuracy bound (`category_top1` held within 0.02 of the fp32 baseline). The gate was applied to every step that touched the model's numerics - including the local INT8 dynamic quantisation path before it was abandoned and the hosted calibrated-INT8 sweep - and held without incident with one near-miss noted in §2ii (the preprocess change at min cosine 0.999767 vs the 0.9999 threshold). The point of the gate is not to track accuracy as a metric - accuracy was held constant by design - but to ensure that a speed win that quietly degraded the model would have been caught as a negative result rather than booked as a positive one.

Raw measurements have been omitted from the final submission for brevity, but were kept under `evals/qps/` (system runs under `system-prod-*`, model runs under `model-prod-*`); each run recorded the config hash, environment, matrix point, and per-run distribution alongside the median.

## 4. What would be worth exploring further

This pass left several questions open by design. The five below are the ones whose answers would most change the operating picture.

1. The round-trip decomposition was done with a controlled probe (n=10) that holds workload constant. The resulting 831 ms bs=24 service time is what feeds the `6 × 24 / 0.831 ≈ 173 QPS` ceiling - but steady-state under live load may differ from the probe (warm-pool dynamics, container scheduling jitter, request-size variance). A direct measurement of steady-state per-call service time during a sustained system bench, broken down by the same layers the probe used, would either confirm the ceiling or surface where the difference lives.

2. Local measurement showed the synchronous (blocking) client contract pins mean batch size at ~8 regardless of total concurrency. Whether the same dynamic binds at the prod operating point is not directly confirmed - prod has a higher absolute throughput and a different service-time/arrival-rate balance. The cleanest unconstrained measurement would be a pipelined client model (multiple in-flight requests per user, or a non-blocking publish/subscribe submission pattern) at otherwise identical settings. If batch fill grew toward 24, both the GPU-side ceiling and the achievable system QPS would lift materially.

3. No measurement covered the hosted-realistic case of the embed pipeline running alongside the LLM-driven `/chat` pipeline - same GKE cluster, shared CPU and network. Under realistic dialogue-paced traffic the embed endpoint sees only ~2–3 requests per second (LLM-bound at ~1 QPS turn rate), so the 157 QPS figure is a burst-capacity number rather than a steady-state one. Contention under combined load - batcher base64 decode competing with LLM streaming for CPU, network bandwidth shared with the conversation channel - has not been characterised.

4. The deployment runs in a single region; the round-trip cost characterised here is regional. A geographically distributed user base would add user→cluster latency that dwarfs everything inside the optimisation envelope above (transcontinental round-trip ~100–150 ms one-way). At that point the marginal value of further system-QPS work falls off sharply - most of the response time is on a wire the cluster does not control. The natural response would be edge regions for the latency-sensitive path (image upload + initial response) with the heavy compute centralised; that is a topology decision, not a tuning decision, and lives in a different design conversation.

5. The A10G was selected as the cheapest Modal tier with sufficient VRAM and memory bandwidth for ViT-base at bs=24 - a catalog running from T4 (~$0.59/hr) and L4 (~$0.80/hr) at the low end, through L40S (~$1.95/hr, Modal's general-inference recommendation) and A100, up to H100 (~$3.95/hr) at the high end. Faster tiers attack only the ~9% of response time that is actually GPU compute; the round-trip dominates, so cost-fit outweighs raw-compute-fit at this scale. The full rationale and cost spectrum are in ARCHITECTURE.md §5C. A formal cost-per-1k-embeddings ablation across configurations - comparing tiers and the always-warm `min_containers=5` posture to a partial cold-start posture - would let the throughput trade-off be weighed against the operating bill rather than treated as orthogonal.

---

The largest single lever in this work - the ~58% round-trip reduction from removing the internal Modal hop - came from instrumenting the system, not tuning it. The result that closed the GPU compute chapter - calibrated INT8 measured to add no advantage at any batch size - came from a cheap decisive diagnostic (a batch-size sweep) that took an afternoon and cost essentially nothing in GPU time. The local series ended not from running out of optimisation ideas but from running out of measurements that pointed at something software-fixable.

In each case, the high-value move was a measurement that *changed what was being optimised*, and the optimisations followed from what the measurements revealed. In a system already engineered competently, the next big throughput win usually does not come from another lever applied to the same model - it comes from a measurement that surfaces a new constraint, or from accepting that the constraint that remains is structural and acting on it at a different layer.
